## ✨ Future enhancements (planned-later upgrades)

_(real upgrades to build once the current system is production-grade)_

- **[pane-lifecycle] automatic idle suspend for completed agents** — park idle/completed panes by preserving scrollback/session metadata and killing the live process tree. Build after the shared Ruflo HTTP and explicit lazy-resume path is stable, because automatic suspension needs clear pinned/keep-alive semantics.
- **[pane-lifecycle] pinned keep-alive policy for remote/autopilot agents** — mark panes that must never be parked by idle cleanup, especially Telegram/remote/autopilot lanes. Build when idle suspend is implemented.
- **[pane-lifecycle] Pane Hibernate: pause/resume-like RAM offload for inactive finished panes** — make the current "inactive/blurred/resuming" pane concept into a real memory feature: keep the pane visible as a cheap frozen shell, but stop the provider/MCP process tree so idle Claude/Codex panes no longer keep 300 MB-1 GB resident.
  Proposed behavior: add a `sleeping`/`hibernated` pane runtime state that preserves `sessionId`, provider, cwd/worktree, branch, runtime profile, external provider session id, scrollback snapshot, last RSS/process breakdown, and the visible pane card. On sleep, stop the PTY process tree and dispose or downgrade the renderer xterm cache; on resume, respawn the provider through the existing resume path and restore the pane to `running`.
  Why it matters: CSS blur/minimize only changes presentation and does not reduce Claude/MCP RSS. Raw OS pause (`SIGSTOP`/`SIGCONT`) is tempting but does not reliably free memory and may break network streams/MCP children. Hibernate is the reliable RAM win because it removes the live process tree until the user needs the pane again.
  Expected impact: in an 8-pane workspace where idle Claude panes average ~500 MB RSS, hibernating 6 finished panes can reduce provider-process RSS from roughly ~4 GB to roughly the 2 active panes plus lightweight pane metadata/scrollback. MCP-heavy panes benefit more because child MCP servers are terminated too.
  Likely touchpoints: `app/src/main/core/pty/registry.ts:401` and `app/src/main/core/pty/registry.ts:411` already expose tree-aware kill/stop; add a hibernate-specific stop path that persists a sleeping state instead of treating the pane as removed. `app/src/renderer/lib/terminal-cache.ts:31` and `app/src/renderer/lib/terminal-cache.ts:367` distinguish parking vs destroy; sleeping panes should probably destroy live xterm/subscriptions after persisting scrollback, while normal workspace switches still park. `app/src/main/core/pty/resume-launcher.ts:460` already supports subset resume by session id; reuse/extend it for explicit pane wake. `app/src/renderer/features/command-room/CommandRoom.tsx:294` and `app/src/renderer/features/command-room/CommandRoom.tsx:301` currently only remove or stop; add Sleep/Wake actions and a header badge.
  Safety policy: manual "Sleep pane" first; later auto-sleep only when there is a clear finished/idle signal, no active prompt, no recent output, low CPU, and the pane is not pinned keep-alive. Never auto-sleep remote/autopilot/Telegram or explicitly pinned panes. On wake failure, keep the frozen pane and show a resumable error rather than deleting the session.
  Trigger: build after the current RAM brake and strict MCP launch defaults have been dogfooded, and after we can reliably persist/restore scrollback for sleeping panes. Effort: L.

## 🆕 New ideas (untriaged)

_(raw ideas land here; promote to ROADMAP.md once scoped into a phase)_

- **[mcp] detect repeated stdio MCP starts under one Codex/Claude pane** — a single pane can accumulate multiple `@claude-flow/cli mcp start` descendants. Investigate whether this is Codex plugin behavior, config reload behavior, or a leaked child cleanup issue once diagnostics can show child process history.

- 🐞 **[medium] `+ Pane` dead on a resumed/restored workspace — swarm comes back `paused`** — after closing the app and reopening (workspaces auto-resumed), the restored workspace's swarm has `status !== 'running'`, so `+ Pane` is gated with the pill *"Swarm is paused — resume it to add panes"* and you cannot add a pane without manually resuming the swarm first. Gate: `app/src/renderer/features/command-room/AddPaneButton.tsx:74-76` (`if (activeSwarm.status !== 'running') → disabledReason`). Root question to settle when scoped: should the workspace-restore/boot path set the swarm back to `running` when its panes are live, OR should `addPane()` auto-resume a paused swarm on click (it already auto-creates one when none exists, `AddPaneButton.tsx:201`)? Investigate the boot resume/swarm-load path that leaves `swarm.status='paused'` (swarm loader in `use-live-events.ts` / session-restore). Severity: medium (workaround: manually resume the swarm). Effort: S–M.

- 🐞 **[high] Jorvis `launch_pane` tool spawns panes that never appear in the Command Room** — Jorvis reports success and even names the new panes (codex/claude builders), but the grid keeps showing only the pre-existing pane. Root cause = **a missing renderer-refresh signal**: the assistant's `launch_pane` MCP tool spawns the PTY + DB rows via `executeLaunchPlan` but **emits no event** (`app/src/main/core/assistant/tools.ts:283-300` — returns `sessionIds`/`sessions`, no broadcast). The Command Room only adds spawned panes to the visible grid when it receives `assistant:dispatch-echo` (`app/src/renderer/features/jorvis-assistant/use-jorvis-dispatch-echo.ts:35` → refetch `panes.listForWorkspace` + `ADD_SESSIONS` at :76-80). The controller's `dispatchPane`/`dispatchBulk` paths DO emit that echo (`app/src/main/core/assistant/controller.ts:466` and `:620`), so panes launched via those show up — but the bare `launch_pane` tool (the one the model actually calls) doesn't. So the backend succeeds (worktree + session insert + PTY spawn) while the renderer never learns → panes invisible. Sibling-twin trap: `launch_pane` is the un-echoed twin of `dispatchPane`/`dispatchBulk` [[grep-sibling-call-sites]]. Fix direction: (A) emit `assistant:dispatch-echo` per session from the `launch_pane` handler — needs `emit` threaded into `ToolContext` (`tools.ts` ~line 39-92 has no `emit` today); OR (B) route the tool through the existing `dispatchPane`/`dispatchBulk` controller path that already echoes correctly (cleaner — single emit site). Confirm no other tool consumer depends on the bare path. Also sanity-check swarm association: if `launch_pane` spawns a swarm-less PTY, a grid that iterates swarm-agent rosters wouldn't show it either. Severity: high (core Jorvis orchestration silently no-ops in the UI). Effort: S–M. Found 2026-06-10 testing the `feat/bsp-pane-tiling` branch.

- 🐞 **[medium] dropping a screenshot into a terminal pane inserts the file PATH, not the image** — drag/drop (or paste) a screenshot onto a pane and the terminal receives the OS file path (a `…/var/…`-style reference), never the actual image, so image-capable CLI agents (Claude Code, Codex) can't see it. Root cause: the terminal host has **no drop/paste interception** — `app/src/renderer/features/command-room/Terminal.tsx` has no `onDrop`/`onPaste` handler (only a resize comment at `:193`), so the drop falls through to xterm's default text-insert of the dragged path. (`PaneShell.tsx:244` sets `dataTransfer.dropEffect='copy'` but that's the pane drag-reorder, not image upload.) Fix direction: intercept image-file drop/paste at the terminal host; for providers flagged image-capable, hand the image to the CLI via its paste-image path (e.g. write the image to the system clipboard in the format the CLI reads, then trigger its image-paste, or base64 over the provider's protocol) — gate on a provider capability flag; for non-image providers keep the path but show a clear affordance. Investigate what Claude Code / Codex expect for pasted images on macOS. Severity: medium. Effort: M.

- 🐞 **[medium] no Copy/Paste on right-click inside a terminal pane** — select text in a pane and right-click → the Radix context menu shows Reveal / Open Shell / Stop / Remove but **no Copy** (and no Paste), and you can't copy the selection. Root cause is twofold: (1) `app/src/renderer/features/command-room/PaneShell.tsx:442-582` wraps the pane body in a Radix `ContextMenu` whose items (`:534-581`) omit Copy/Paste — and because Radix captures the right-click, xterm's own native right-click-copy never fires either; (2) there's **no xterm clipboard wiring** — `app/src/renderer/lib/terminal-cache.ts` sets no `copyOnSelect` / `rightClickSelectsWord` and pulls in no `@xterm/addon-clipboard`, and nothing anywhere calls `term.getSelection()`/`hasSelection()`. Fix direction: add **Copy** (enabled when the pane's xterm `hasSelection()`) + **Paste** items to the ContextMenu — Copy → `term.getSelection()` → `navigator.clipboard.writeText`; Paste → read clipboard → write to the PTY. Needs a ref to the pane's terminal instance to read selection (thread it from `terminal-cache`/Terminal up to PaneShell). Optionally enable `copyOnSelect` for convenience. Sibling note: keep the menu's existing Reveal/Stop/Remove items. Severity: medium (core terminal usability gap). Effort: S–M.
