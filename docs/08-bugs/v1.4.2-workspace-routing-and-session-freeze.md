# v1.4.2 dogfood — Workspace routing regression + perceived session freeze

> Two bugs filed from user dogfood of v1.4.1 (commit `6e635db`, 2026-05-17).
> Both are user-visible regressions of work that was supposed to ship in v1.3.3 (Bug A) and v1.2.7 (Bug B).
> Investigation done read-only — no code edits in this entry.

---

## Bug A — Sidebar workspace click stuck on Settings after Settings visit

### Symptoms / repro
User report verbatim: *"When clicked to settings and tried to switch back to workspace via clicking to the Workspace from left panel not working, need to go to menu and click Command Room."*

Steps that reproduce:
1. Open a workspace; Command Room renders panes as expected.
2. Click the gear/Settings entry (top-bar room switcher) — `state.room` becomes `'settings'`, SettingsRoom mounts.
3. Click the same (or any open) workspace row in the left sidebar.
4. **Expected**: workspace re-activates and the user lands back on Command Room.
5. **Actual**: workspace re-activates but the user stays on Settings — only the top-bar room dropdown can route them back to Command Room.

### Suspected code

| File | Line | Why it matters |
|---|---|---|
| `app/src/renderer/features/sidebar/Sidebar.tsx` | 118–120 | `onPick` dispatches **only** `SET_ACTIVE_WORKSPACE_ID`. The v1.3.3 commit `66b4fa6` REMOVED the explicit `SET_ROOM='command'` follow-up dispatch from this handler (kept it on `onOpenPersisted` at line 62) on the theory the reducer would route via the per-workspace room recall. |
| `app/src/renderer/app/state.reducer.ts` | 191–192 | `SET_ACTIVE_WORKSPACE_ID` reads `state.roomByWorkspace[action.workspaceId]`; if a saved room exists AND it is not `'workspaces'`, the reducer uses it directly. `'settings'` passes that guard. |
| `app/src/renderer/app/state.reducer.ts` | 91–102 | `SET_ROOM` persists per-workspace room via `roomByWorkspace[wsId] = action.room` for every room except `'workspaces'`. Settings is NOT filtered, so visiting Settings writes `roomByWorkspace[currentWs] = 'settings'`. |
| `app/src/renderer/app/state.types.ts` | 21–39 | `RoomId` union includes `'settings'` alongside per-workspace rooms — there is no "global room" sentinel distinguishing settings from workspace-scoped rooms. |

### Root-cause hypothesis (one sentence)
v1.3.3's reducer-level fix (commit `66b4fa6`) routes `SET_ACTIVE_WORKSPACE_ID` to whichever room was saved in `roomByWorkspace[wsId]` and only special-cases `'workspaces'`; because `SET_ROOM` happily persists `'settings'` into the per-workspace map (state.reducer.ts:99 only filters `'workspaces'`), clicking a workspace tile after a Settings visit replays the user back into Settings instead of Command Room.

### Likely fix shape (do not implement here)
Two coherent options — pick one:
- **Filter at the writer**: extend the `SET_ROOM` guard at state.reducer.ts:99 so `'settings'` (and any future global room) is excluded from `roomByWorkspace` — settings is not a workspace-scoped surface.
- **Filter at the reader**: extend the guard at state.reducer.ts:192 so a saved room of `'settings'` falls through to `'command'` the same way `'workspaces'` already does. Cheaper but leaves stale entries in the persisted map.

Either fix should be paired with a reducer test case (`state.reducer.test.ts`) covering "user was on Settings → SET_ACTIVE_WORKSPACE_ID routes them to Command Room, not Settings."

### Verification approach
- Reducer unit: dispatch sequence `SET_ACTIVE_WORKSPACE_ID(wsA) → SET_ROOM('settings') → SET_ACTIVE_WORKSPACE_ID(wsA)`; final `state.room` must equal `'command'`.
- Manual smoke: repro steps above on the v1.4.2 build; pane grid must render after the sidebar click without an intermediate top-bar room hop.
- Cross-check: visiting other "global-feeling" rooms (e.g. Skills today is per-workspace; Settings is the only true global) and switching workspaces should not bounce the user back into the prior room category.

## Bug B — Sessions feel frozen until clicked

### Symptoms / repro
User report verbatim: *"feels like sessions getting frozen, and need to click to them and wait until they resume which I want them operate like normal terminal regardless of I'm in that workspace or somewhere else in the app."*

User's stated expectation: PTY panes behave like a normal terminal multiplexer — they keep displaying output regardless of which room or workspace is focused.

Steps that reproduce:
1. Open workspace A with one or more active CLI panes (claude/codex/etc.) that are emitting output.
2. Navigate to any non-Command room (Settings, Memory, Browser, Sigma) or switch to workspace B.
3. Return to workspace A's Command Room.
4. **Expected**: panes show every byte of output that arrived while away, with no perceptible delay before the live stream resumes.
5. **Actual**: each pane appears static / blank for a noticeable window after the room or workspace change; clicking the pane (or otherwise giving it time) makes it "come back to life." Older output that arrived while away may be missing.

### Suspected code

| File | Line | Why it matters |
|---|---|---|
| `app/src/renderer/app/App.tsx` | 91–142 | `RoomSwitch` returns a different subtree per `state.room`. Going to Settings unmounts `<CommandRoom>` entirely, which cascades unmount into every `<SessionTerminal>`. CommandRoom is the only eager case (line 106); every other room is lazy + Suspense-fallback. |
| `app/src/renderer/features/command-room/Terminal.tsx` | 267–278 | Cleanup disposes xterm, unsubscribes from the live pty data bus, disconnects the ResizeObserver. On remount everything is rebuilt from scratch. |
| `app/src/renderer/features/command-room/Terminal.tsx` | 152–167 | Mount sequence is **await `rpc.pty.snapshot(sessionId)` → write snapshot → THEN `subscribePtyData`**. The IPC + write happens inside an async IIFE; the live subscription only attaches *after* the snapshot resolves. Any output emitted by the PTY between the snapshot read and the subscription attach is dropped from the renderer's view (already filed as v1.2.7 open risk R-1.2.7-1, see `docs/03-plan/WISHLIST.md` "Terminal.tsx mount race"). |
| `app/src/main/core/pty/registry.ts` (~line 105–163) | — | Per the v1.2.7 plan, the ring buffer is bounded at ~64KB per session. Output emitted while no renderer was attached past 64KB silently rotates out — when the user returns, the snapshot replay only shows the most recent slice. For a chatty agent (Claude streaming, Codex tool-use loops) this can blow the budget inside seconds. |

### Root-cause hypothesis (one sentence)
v1.2.7 shipped the ring-buffer-replay model deliberately as Approach 1 (cheaper than Approach 2 — keep xterm alive via React 19 `<Activity>` / a renderer-side terminal cache); the PTY processes themselves DO keep running across room/workspace switches, but the xterm instance is unmounted, so on return each pane has to (a) re-instantiate xterm, (b) round-trip an async `pty.snapshot` IPC, (c) write the buffered bytes, (d) only then attach the live data bus — producing the "frozen until clicked" feel, compounded by the 64KB ring-buffer cap losing older output for chatty agents.

### Likely fix shape (do not implement here)
Two layers, both already queued in WISHLIST:
- **Quick win (XS, ~5 min)**: WISHLIST entry "Terminal.tsx mount race" — attach the `subscribePtyData` listener BEFORE awaiting `rpc.pty.snapshot`, buffer incoming bytes into a local array, replay snapshot then drain the local buffer. Closes R-1.2.7-1's 1–5 ms IPC drop window and removes one source of the "frozen" feel on cold remount.
- **Real fix (L)**: v1.2.7 follow-up explicitly named in `docs/03-plan/v1.2.7-multi-workspace-state-preservation.md` line 17, 38, and `docs/08-bugs/BACKLOG.md` line 40 — true xterm instance preservation. Either (a) React 19 `<Activity mode="hidden">` wrapping the CommandRoom subtree so the workspace tree stays mounted but paused when the room is not active, or (b) a renderer-side terminal cache (`useRef`/Zustand store of live `Terminal` instances) re-attached via `terminal.open(newDiv)` on remount. This is what makes the experience match the user's "normal terminal multiplexer" expectation — zero-latency switching with no buffer loss. The ring-buffer cap also wants a bump (64KB → 256KB + LRU eviction was already flagged as risk R-1.2.7-1 mitigation).

Pairing well-defined success criteria for the real fix: switching room or workspace must not unmount xterm; the PTY pid count before and after a switch is unchanged (already covered by `tests/e2e/multi-workspace.spec.ts`); a Playwright assert that the new last-line byte count after a 30 s heavy-output run equals the byte count observed if the user never left the room.

### Verification approach
- Reproduce on v1.4.1: open a workspace with a Claude pane streaming long output, switch to Settings for ≥ 10 s, switch back; observe the "frozen" gap and any lost lines vs. terminal scrollback.
- After fix: rerun the same sequence; pane must continue to render live without a perceptible gap and without dropped output.
- Regression coverage: extend `tests/e2e/multi-workspace.spec.ts` with a "no buffer loss across room switch" assertion (drive a known number of bytes through a shell pane, switch rooms for N seconds, switch back, assert `term.buffer.length` matches expectation).
