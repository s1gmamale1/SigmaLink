[Heading 1] Executive Summary
[Normal] This research report provides a comprehensive analysis of how to build a unified workspace environment for orchestrating multiple AI coding agents — similar to BridgeMind's BridgeSpace product. The core innovation requested is a dynamic multi-provider system where any CLI agent (Claude Code, Codex, Gemini CLI, OpenCode, Aider, etc.) can be launched in parallel terminals, connected to each other, and delegated tasks by a central orchestrator AI.
[Normal] After evaluating 15+ open-source tools and frameworks across the rapidly evolving agent orchestration landscape, this report identifies Emdash as the strongest foundation for transformation, with dux and Claude Squad as viable alternatives for terminal-first workflows. The report provides a complete architecture blueprint, comparative analysis matrix, and a 16-week implementation roadmap.
[Normal] Key Findings: Emdash (Apache-2.0, Electron) is the strongest foundation for a BridgeSpace alternative. Dux (Go TUI) offers the best terminal-first experience. Bernstein provides the most sophisticated orchestrator architecture. The combination of Emdash + a custom orchestrator layer inspired by Bernstein patterns represents the optimal path to a production-quality multi-agent workspace with dynamic provider support.
[Heading 1] BridgeSpace Feature Analysis
[Normal] BridgeSpace by BridgeMind is a commercial Agentic Development Environment (ADE) that provides a desktop workspace for orchestrating up to 16 AI coding agents in parallel. It represents the current state-of-the-art in multi-agent workspace design and serves as the primary reference for the desired system.
[Heading 2] Core Architecture Overview
[Normal] •  Desktop Application: Cross-platform Electron-based app (macOS, Windows, Linux)
[Normal] •  Terminal-First Design: Multiple terminal panes running real CLI agents via PTY
[Normal] •  Git Worktree Isolation: Each agent gets its own isolated working directory
[Normal] •  Multi-Room Workspace: Command Room (terminals), Swarm Room (parallel agents), Review Room (human oversight)
[Normal] •  BridgeMemory: Local-first knowledge graph with 12 MCP tools for shared agent memory
[Normal] •  Kanban Task Board: Integrated project management with task-to-agent assignment
[Heading 2] Key Differentiators
[Normal] What separates BridgeSpace from simpler multi-terminal managers is its orchestration intelligence. The system includes an orchestrator CLI AI that can:
[Normal] •  Decompose high-level tasks into sub-tasks suitable for parallel execution
[Normal] •  Delegate sub-tasks to specialized agent instances with appropriate context
[Normal] •  Monitor agent progress and verify outputs before merging
[Normal] •  Maintain shared context through BridgeMemory (wikilink-based knowledge graph)
[Normal] •  Support agent-to-agent communication for collaborative problem-solving
[Heading 1] Open Source Foundation Candidates
[Normal] The open-source ecosystem for multi-agent orchestration has matured rapidly in 2025-2026. The following tools were evaluated as potential foundations, ranked by their suitability for building a BridgeSpace-style unified workspace with dynamic multi-provider support.
[Heading 2] Tier 1: Emdash (Primary Recommendation)
[Normal] Emdash is the strongest open-source foundation for building a BridgeSpace alternative. It is an Electron desktop app (TypeScript/React) with native multi-provider support, SSH remoting, ticket integration, and active YC-backed development.
[Normal] Emdash (generalaction/emdash on GitHub, 4,300+ stars, Apache-2.0) is an open-source Agentic Development Environment from Y Combinator W26 graduates. It directly competes with BridgeSpace and shares the same fundamental architecture.
[Heading 3] Key Specifications
[Heading 3] Supported Agents (Auto-Detected)
[Normal] Claude Code, OpenAI Codex, Gemini CLI, OpenCode, Amp, Droid (Factory CLI), Hermes Agent, Qwen Code, Cursor CLI, GitHub Copilot CLI, Aider, and others. Adding a new provider requires only a configuration entry — no code changes.
[Heading 3] Strengths for Transformation
[Normal] •  Already an Electron desktop app with multi-pane terminal UI
[Normal] •  Provider abstraction layer makes adding new agents trivial
[Normal] •  Built-in git worktree management and diff review
[Normal] •  Active, well-funded development (YC W26)
[Normal] •  Commercial-friendly Apache-2.0 license
[Heading 3] Gap Analysis
[Normal] The primary gap in Emdash is the orchestrator intelligence layer. Emdash explicitly does not include agent-to-agent delegation or an orchestrator AI that can decompose tasks and delegate. This is the key feature to build on top of the Emdash foundation.
[Heading 2] Tier 2: Dux (Terminal-First Alternative)
[Normal] Dux is a Go-based Terminal UI application that runs multiple AI coding agents side-by-side in git worktrees. It is the closest open-source tool to the requested orchestrator concept, supporting any CLI provider through configuration.
[Normal] Dux (patrickdappollonio/dux on GitHub) is a terminal-native orchestration layer written in Go. Its core philosophy: "No protocol layers. No adapters. No JSON-RPC. Just real CLIs in real terminals."

[Normal] •  Command Palette: Fuzzy-searchable access to all actions
[Heading 2] Tier 3: Claude Squad
[Normal] Claude Squad (smtg-ai/claude-squad on GitHub, 6,800+ stars, AGPL-3.0) is a Go-based terminal app for managing multiple AI coding agents with worktree isolation. It focuses on human-in-the-loop supervision.
[Normal] •  Agents: Claude Code, Codex, Aider, Gemini, OpenCode, Amp
[Normal] •  Isolation: Git worktrees + tmux sessions per agent

[Heading 2] Tier 4: Bernstein (Orchestrator-Focused)
[Normal] Bernstein (chernistry/bernstein on GitHub, Apache-2.0) is the most architecturally interesting orchestrator. It implements a full planning-to-merge pipeline with deterministic scheduling.
[Normal] •  Pipeline: Goal -> LLM Planner -> Task Graph -> Orchestrator -> Agents (parallel) -> Janitor (verify) -> Git merge
[Normal] •  Deterministic scheduling with zero LLM tokens spent on coordination
[Normal] •  Supports 30+ CLI agents including Claude Code, Codex, Gemini CLI
[Normal] •  Janitor agent runs lint, type checks, and tests before merge
[Normal] •  Local HTTP task server for agent progress reporting

[Heading 1] Architecture Blueprint
[Normal] Based on the research, the recommended architecture combines Emdash as the Electron desktop foundation with a custom orchestrator layer built on proven patterns from Bernstein, dux, and Microsoft Agent Framework.
[Heading 2] System Architecture
[Normal] The system consists of five core layers:
[Heading 3] 1. Desktop Shell (Electron + React)
[Normal] •  Multi-pane terminal layout using xterm.js + node-pty
[Normal] •  Room-based navigation: Command Room, Swarm Room, Review Room
[Normal] •  Kanban task board with drag-and-drop agent assignment
[Normal] •  Diff viewer with syntax highlighting and inline commenting
[Normal] •  Activity timeline for tracking agent actions
[Heading 3] 2. Worktree Manager
[Normal] •  Creates isolated git worktrees per agent session
[Normal] •  Automatic branch naming: orchestrator/{agent-type}/{task-id}
[Normal] •  Port allocation per worktree to prevent conflicts
[Normal] •  Environment variable injection for provider API keys
[Heading 3] 3. Agent Provider Abstraction
[Normal] •  Plugin-based provider system (inspired by Emdash)
[Normal] •  Each provider defines: command, args, resume_args, oneshot_args
[Normal] •  Auto-detection: scan PATH for known CLI binaries
[Normal] •  Provider configuration in TOML/JSON format
[Heading 3] 4. Orchestrator Engine
[Normal] •  LLM-based task decomposition using structured output (JSON mode)
[Normal] •  Hierarchical delegation: Orchestrator -> Sub-Agents -> Workers
[Normal] •  Verification loop: Execute -> Verify -> Retry if failed
[Normal] •  Deterministic scheduling (Bernstein pattern): zero token cost for coordination
[Normal] •  Shared memory via MCP tools (create_memory, search_memories)
[Heading 3] 5. Communication Layer
[Normal] •  MCP (Model Context Protocol) for agent-to-tool communication
[Normal] •  Local HTTP server for agent status/progress reporting
[Normal] •  Message bus for inter-agent coordination
[Normal] •  SQLite for persistent state, filesystem for shared artifacts
[Heading 2] The Orchestrator Layer
[Normal] The orchestrator is the key differentiator. It should implement the hierarchical manager-worker pattern with verification gates:
[Normal] // Orchestrator execution loop
for each subtask:
  1. EXECUTE: Delegate to appropriate specialist agent
  2. VERIFY: Ask verifier agent 'Is this correct enough?'
  3. If NO: Re-delegate with verifier feedback -> goto 1
  4. If YES: Subtask complete -> next subtask

// Delegation contract
{ intent, inputs, constraints, success_criteria, assigned_agent }

[Heading 1] Technical Deep Dive
[Heading 2] Git Worktree Isolation
[Normal] Git worktrees are the consensus isolation primitive across all evaluated tools. Each agent session gets its own working directory and branch:
[Normal] // Create worktree for agent session
git worktree add -b agent/task-001 ../project-task-001 main

// Each agent runs in its own directory
// No file conflicts between parallel agents
// Merge results via standard git merge/PR workflow

// Cleanup when done
git worktree remove --force ../project-task-001
git branch -D agent/task-001
[Heading 2] PTY Terminal Emulation
[Normal] node-pty (a native Node.js module) creates pseudo-terminals that allow interactive CLI agents to run as if in a real terminal. Combined with xterm.js in the renderer process, this provides full terminal emulation in the Electron UI.
[Normal] •  Full TTY emulation: colors, cursor positioning, escape sequences
[Normal] •  Bidirectional I/O: read and write to the terminal process
[Normal] •  Process control: resize, kill, and monitor
[Normal] •  Cross-platform: Linux/macOS (unix PTY), Windows (ConPTY)
[Heading 2] MCP Protocol Integration
[Normal] The Model Context Protocol (MCP), developed by Anthropic, standardizes how agents access external tools and context. It enables the orchestrator to expose shared memory, file system access, and inter-agent communication as tools that any agent can call.
[Normal] •  MCP servers expose: tools, resources, and prompts
[Normal] •  Agents call tools via structured JSON-RPC
[Normal] •  Shared memory tools: create_memory, search_memories, find_backlinks
[Normal] •  Enables any MCP-compatible agent to participate in the workspace
[Heading 1] Conclusions & Recommendations
[Normal] Building a BridgeSpace-style unified workspace with dynamic multi-provider orchestration is achievable today using existing open-source foundations. The ecosystem has matured significantly, with multiple viable starting points.
[Heading 3] Primary Recommendation
[Normal] Fork Emdash and build the orchestrator layer on top. Emdash provides the Electron desktop shell, multi-provider agent abstraction, worktree management, and SSH support — all under Apache-2.0. The missing orchestrator intelligence can be added incrementally without rewriting the foundation.
[Heading 3] Alternative: Greenfield Build
[Normal] If maximum flexibility is required, build from scratch using: Electron + React (desktop shell), node-pty + xterm.js (terminals), SQLite (state), and a custom orchestrator engine inspired by Bernstein's deterministic scheduling. This approach requires more effort but eliminates any architectural constraints from an existing codebase.
[Heading 3] Key Success Factors
[Normal] •  Provider abstraction must be config-driven, not hardcoded
[Normal] •  Git worktrees are essential for conflict-free parallel execution
[Normal] •  Verification gates prevent error propagation in agent chains
[Normal] •  Human-in-the-loop design maintains developer control
[Normal] •  MCP integration maximizes interoperability with the agent ecosystem
[Normal] The window of opportunity is open. The tools, patterns, and protocols needed to build this system are all production-ready. The differentiator will be execution quality and the sophistication of the orchestrator intelligence layer.
