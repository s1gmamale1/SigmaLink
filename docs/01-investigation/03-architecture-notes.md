# 03 - Architecture Notes

This is a defensible review of the Phase 1 architecture: what works, what is fragile, and what needs to change before Phase 2 lands. Citations are file:line.

---

## Overall shape

- **Process model.** Electron main hosts the typed RPC router (`app/src/main/rpc-router.ts`) and owns SQLite, node-pty, and worktree state. Renderer is a single `BrowserWindow` (`app/electron/main.ts:15-49`) with `contextIsolation: true`, `nodeIntegration: false`, `sandbox: false`. A thin preload bridge (`app/electron/preload.ts`) exposes one generic `invoke` plus typed events.
- **RPC layer.** Channel naming is `<namespace>.<method>`. Both sides build a Proxy at runtime (`app/src/shared/rpc.ts:28-44`) so callers write `rpc.workspaces.launch(plan)` and the Proxy translates to `ipcRenderer.invoke('workspaces.launch', plan)`. Envelope is `{ ok: true, data } | { ok: false, error }`. Renderer unwraps via `app/src/renderer/lib/rpc.ts:7-18`.
- **State.** Renderer state is plain `useReducer` + Context (`app/src/renderer/app/state.tsx`). No Redux, no Zustand. Reducer actions are explicit and small.
- **Persistence.** SQLite via better-sqlite3 + Drizzle ORM (`app/src/main/core/db/{client,schema}.ts`). Schema is generated by hand-rolled CREATE TABLE statements at boot; Drizzle migrations are deferred.
- **Spawn surface.** Two paths: long-running interactive PTYs go through `node-pty` (`app/src/main/core/pty/local-pty.ts`) and short-lived utilities go through `child_process.spawn` with `shell: false` (`app/src/main/lib/exec.ts`).

## Strengths (defensible in review)

### S1 - Single bridge, typed end-to-end
The preload exposes a deliberately dumb generic `invoke`; the type contract lives in `app/src/shared/router-shape.ts`. Renderer and main both depend on the same `AppRouter` interface so refactoring a controller signature surfaces in both. This is materially better than a shared-strings IPC channel list. (`router-shape.ts:13-63`).

### S2 - Subscribe-then-replay ordering for PTY history
`Terminal.tsx:75-93` documents the ordering rule: register the live `pty:data` listener first, then call `rpc.pty.subscribe` to get the snapshot. Because `subscribe` returns the buffer atomically and live data continues to flow into the listener, no race between replay and live can drop bytes. The discipline is correct; the implementation has small caveats (P2 entries) but the architecture is sound.

### S3 - Provider registry as data
`app/src/shared/providers.ts` keeps providers as a flat array of plain records. The same data is consumed by main (probe + spawn) and renderer (UI cards), with a tiny pure helper `findProvider`. Adding a provider is a one-line change to the array. No reflection, no DI container, no factory class.

### S4 - Worktree pool is functional + scoped per repo
`WorktreePool` (`app/src/main/core/git/worktree.ts`) hashes the repo root into a stable per-repo directory under `userData/worktrees/<sha1[0:12]>` and gives each session a per-branch subdirectory. Branch names are sanitised (`sanitizeBranchSegment`). Removal is idempotent (`fs.existsSync` guard).

### S5 - Argument-array exec everywhere except runShellLine
`exec.ts:21-71` uses `spawn(command, args, { shell: false, windowsHide: true })`. No shell interpolation, no shell-injection vector via cwd. The single tokenised path (`runShellLine`) is the exception (and is currently exposed but unused — see P1-RUN-SHELL-TOKENISER).

### S6 - Ring buffer is bounded and per-session
`RingBuffer` (`app/src/main/core/pty/ring-buffer.ts`) caps each session at a default 256 KiB and trims FIFO. Snapshot is a single `chunks.join('')`. Simple, debuggable, and decoupled from xterm.

### S7 - DB schema is intentionally narrow for Phase 1
Three tables (`workspaces`, `agent_sessions`, `kv`) cover the Phase 1 surface. Foreign keys are documented in code (`workspaceId` references `workspaces.id`) without being enforced at the DB level — acceptable for Phase 1 because writes are mediated by the launcher.

### S8 - Reducer state shape is thin and immutable
The renderer state shape (`app/src/renderer/app/state.tsx:18-25`) holds `workspaces`, `activeWorkspace`, `sessions`, `activeSessionId`, `room`, `ready`. ADD_SESSIONS dedupes via Map. SET_ACTIVE_WORKSPACE auto-routes the room. Easy to reason about.

## Weaknesses (must address before Phase 2)

### W1 - Spawn path is platform-incomplete
The Windows spawn path only wraps `.cmd/.bat/.ps1` when the command string already contains the extension (`local-pty.ts:41-56`). All npm-installed CLIs are extensionless and bypass the wrapper, breaking every agent on Windows (P0). Same hole exists in `probe.ts` and `runShellLine` for the same reason. The architecture needs a single `resolveForCurrentOS(command)` choke point that all spawn callers go through; right now the responsibility is duplicated and inconsistent.

### W2 - PtyRegistry has no end-of-life
`registry.forget(id)` is defined and never called (`registry.ts:94-100`). Sessions linger forever in memory (with their 256 KiB ring buffers) and the matching DB rows stay `'running'`. There is no UI affordance to close a pane, no main-side janitor that cleans up post-exit. This will surface as a leak the moment a Phase 2 swarm spins up and tears down agents.

### W3 - Failure modes are silent
- Synchronous `nodePty.spawn` throw is uncaught (`local-pty.ts:66-72`).
- Spawn-time CreateProcess failures are reported only as a single data chunk; no `pty:exit` is synthesised, so the renderer keeps the pane "running" forever.
- DB errors in the post-exit listener are swallowed (`launcher.ts:122-124`).
- RPC layer drops stack traces (`rpc-router.ts:177-180`).
The cumulative effect is that Phase 1 has no observability story. A `log.ts` with timestamped output (and forwarding to renderer console) would cost ~30 LoC and pay for itself in every subsequent debug session.

### W4 - Preload exposes generic `invoke`, no allowlist
Any renderer code (or any compromised npm package executing in the renderer) can call any registered channel. The router includes `git.runCommand` (which `child_process.spawn`s arbitrary commands in arbitrary cwds with no further validation). The contract claims sandbox=off is fine because there's no remote content; that is true today but the threat model degrades the moment a Phase 3 web view, an embedded MCP server, or a Skills drag-drop accepts external content. Allowlist preload to the channels enumerated in `router-shape.ts` and reject anything else.

### W5 - Launcher has no transactional rollback
`executeLaunchPlan` (`launcher.ts:33-126`) creates a worktree, spawns a PTY, inserts a DB row, and registers an exit listener — none of it inside a try/catch. A failure mid-loop leaves a partial set of panes and orphaned worktrees on disk. The launch path should treat each pane as a unit with explicit cleanup on failure.

### W6 - Event broadcast is per-window, not per-subscriber
`broadcast` (`rpc-router.ts:20-24`) sends every PTY data chunk to every BrowserWindow. Today there is one window so this is moot. But the renderer's filtering pattern (`if (p.sessionId === sessionId) term.write(...)`) means every Terminal instance in the app receives every other terminal's data. With 16 panes you get a 16x amplification of the IPC traffic per chunk. The right shape is a subscription registry keyed by webContentsId+sessionId (and clean-up on unmount).

### W7 - DB lifecycle has no graceful shutdown
The DB handle is opened on first window load and never closed. WAL mode is forgiving but a `before-quit` handler that calls `rawDb.close()`, kills surviving PTYs, and removes orphaned worktrees would close the loop. Without it, sessions that were "running" at quit stay "running" in the DB forever.

### W8 - Reducer cannot remove sessions
`Action` has no `REMOVE_SESSION`. Once a session is added it persists in `state.sessions` until app reload. Combined with W2 this means every kill-and-relaunch cycle grows the in-memory list.

### W9 - The provider registry duplicates spawn logic in two places
`rpc-router.ts:51-69` and `launcher.ts:33-126` each call `pty.create` with provider data. They both build `args`. They both schedule the initial-prompt write. Today the controller path is unused for launches but it is exposed via RPC, so there are two places where future args/env logic must be kept in sync. Consolidate into one helper.

### W10 - SQL bootstrap drift risk
`client.ts:14-47` hand-rolls the schema in SQL strings. The Drizzle schema in `schema.ts` is the canonical type source. They will drift; the bootstrap SQL is more constrained (e.g. enum check constraints). At minimum: generate the bootstrap from the Drizzle schema or commit to migrations.

## Component-by-component review

### Provider registry (`app/src/shared/providers.ts`)
- Strength: pure data, plain TypeScript record. No side effects.
- Weakness: `command` is overloaded — used for spawn, probe, and altCommands handling diverges. After P0 is fixed, separate fields (`probeCommand`, `spawnTarget`) might be clearer.

### RPC layer (`app/src/shared/rpc.ts`, `electron/preload.ts`, `src/renderer/lib/rpc.ts`)
- Strength: Proxy-based, end-to-end typed via `AppRouter`.
- Weakness: preload allowlist (W4); error stacks lost (W3); per-namespace cache never released.

### PTY local + registry (`app/src/main/core/pty/{local-pty,registry,ring-buffer}.ts`)
- Strength: minimal API surface; ring buffer is correct; subscribe/replay ordering is documented.
- Weakness: spawn is platform-incomplete (W1); no end-of-life (W2); no spawn-time error pathway (W3).

### Worktree pool (`app/src/main/core/git/worktree.ts`, `git-ops.ts`)
- Strength: pure functions, deterministic paths, idempotent removal.
- Weakness: no rollback on launch failure; 5-char random suffix; sanitiser collapses dotted hints.

### DB (`app/src/main/core/db/{client,schema}.ts`)
- Strength: simple, narrow, WAL on, foreign_keys on.
- Weakness: bootstrap SQL drifts from Drizzle schema; no graceful close; no migrations.

### Renderer state (`app/src/renderer/app/state.tsx`)
- Strength: plain reducer, dedupe via Map, idiomatic.
- Weakness: no remove action; effects rely on `window.sigma` being present (handled by error in `rpc.ts:9-10`).

### Renderer features
- Sidebar (`app/src/renderer/features/sidebar/Sidebar.tsx`) is fine; `aria-current` missing (P2).
- Workspace launcher (`app/src/renderer/features/workspace-launcher/Launcher.tsx`) is the most complex Phase 1 component; logic is straightforward, error display is minimal but functional.
- Command Room (`app/src/renderer/features/command-room/CommandRoom.tsx`) is the place that will need most evolution: no per-pane close, no keyboard nav, focus-mode toggles by mutating layout+focusId together.
- Terminal (`app/src/renderer/features/command-room/Terminal.tsx`) implements subscribe/replay correctly; minor robustness gaps (P2 series) around resize debouncing and unicode-safe ring buffer slicing.

## What I would not change

- The decision to keep main strictly in TypeScript and bundle with esbuild (`scripts/build-electron.cjs`) is right; it keeps the dev-build vs. ship-build identical.
- The decision to keep `node-pty` and `better-sqlite3` external in the esbuild bundle is right.
- The `<namespace>.<method>` IPC convention is right; it scales linearly with controllers.
- The `RpcResult<T>` envelope is right (avoids exceptions across IPC).
- The single shared types module (`app/src/shared/types.ts`) is right; types live in one place.

---

## What needs to change before Phase 2

In rough priority order:

1. Fix W1 (Windows spawn) — required for the product to function on Windows.
2. Fix W2 (PtyRegistry forget) and W8 (REMOVE_SESSION) — required because Phase 2 swarms cycle agents.
3. Fix W3 (failure observability) — required for any further work; debugging without logs is throwing darts.
4. Fix W5 (launch rollback) — required because Phase 2 will spawn many panes at once and partial failures must not leak resources.
5. Fix W4 (preload allowlist) — required before any Phase 3 web/MCP work.
6. Fix W6 (per-window event subscription) — soft requirement; bumps performance and unblocks multi-window in Phase 4.
7. Fix W7 (graceful DB close) — required for production builds.
8. Fix W9 (deduplicate spawn logic) — soft requirement; reduces drift risk.
9. Fix W10 (Drizzle migrations or codegen) — required before any schema change.
