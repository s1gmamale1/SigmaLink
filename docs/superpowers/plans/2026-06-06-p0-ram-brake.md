# P0 RAM Brake Implementation Note

Date: 2026-06-06

## Implemented

- Added shared runtime profiles:
  - `ruflo-core` is the default and writes only lightweight Ruflo MCP config.
  - `browser-tools` opts a lane into Browser and SigmaMemory MCP config.
  - `security-tools` and `full-tools` are reserved for explicit future use.
- Threaded `runtimeProfileId` through workspace launch plans, swarm roster materialization, and `+Pane`.
- Added a **Browser tools** toggle in the launcher and `+Pane` menu so heavy browser/memory MCP servers are no longer global defaults.
- Hardened MCP config writing so Browser/SigmaMemory blocks are omitted unless the selected profile allows them.
- Added macOS process-tree inspection/stopping behind `PtyRegistry`.
- Extended Maintenance cleanup dry-runs with live process-tree snapshots and live RSS totals.
- Added an intentional stop-live path for destructive cleanup: live pane process trees can be stopped before pane rows are cleared, while live worktree paths remain fenced from deletion.

## Why This Improves Memory Pressure

The prior launch path could attach Browser/SigmaMemory-capable MCP config to every active agent pane. With 15-20 agents, that multiplies MCP startup surfaces and makes accidental plugin fan-out much more likely. The new default keeps ordinary coding lanes on Ruflo only, then lets the operator opt specific lanes into Browser/SigmaMemory when the task actually needs them.

The cleanup side addresses the second failure mode: completed or abandoned pane trees can remain alive outside the visible UI. Maintenance can now show the live RSS footprint and stop those process trees deliberately instead of only deleting database rows.
