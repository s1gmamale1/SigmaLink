# Plan — v1.2.7: Multi-workspace state preservation (sidebar + scrollback replay + resume reliability)

> User reports that adding/removing/switching workspaces "kills or interrupts the current one" and that session-saving "added before but some change override or removed that feature." Reference screenshot is V3 BridgeMind: left sidebar with WORKSPACES header, "+" button, chevron dropdown, 5 workspace rows each with colored dot + name + pane-count badge + close-X. Switching between them must NOT interrupt running CLI agents in any of them.

## Context — what's actually broken

Three parallel investigations (codebase persistence audit, PTY lifecycle audit, industry pattern research) closed today and reframed the user's report:

**PTYs are NOT being killed on workspace switch today.** The registry (`app/src/main/core/pty/registry.ts:91-204`) is process-wide. `SET_ACTIVE_WORKSPACE_ID` is a pure renderer reducer with no main-process handler — no IPC fires, no `pty.kill` is called. `CommandRoom.tsx:66-77` reads `state.sessionsByWorkspace[activeWorkspaceId]` and is **not keyed by workspaceId**, so the component instance persists.

**But xterm instances DO get unmounted on workspace switch.** When the user switches from workspace A to workspace B, React diffs out the per-pane `<Terminal>` components for A's sessions (because the filtered list changes), unmounts them, and the cleanup at `Terminal.tsx:268-278` disposes the xterm instance + unsubscribes from live PTY data. The PTY itself keeps running in main, output keeps being appended to the registry's `RingBuffer` (line 111, 163), but **the user sees a fresh black terminal with no scrollback** when they return. That is what they're calling "interruption."

**Boot-time persistence is fragile in two specific ways:**
1. `external_session_id` capture (registry.ts:113-160) is a best-effort 100-line scan of the CLI agent's startup output looking for a provider-specific session ID pattern. If the CLI prints its banner late or in a format we don't recognise, the scan times out and the session is stored with `external_session_id = NULL`.
2. `resume-launcher.ts:188` filters resumable rows by `external_session_id IS NOT NULL`. Sessions without it are silently skipped, and `use-session-restore.ts:108` catches the resume RPC error without surfacing a toast — so the user opens the app expecting their workspace back, sees an empty CommandRoom, and concludes "session saving was removed."

**Industry pattern (researcher report)**: VS Code's Pty Host model — PTYs in main with a ring buffer that keeps draining even when no renderer is attached, replay-on-reattach IPC, xterm instances kept alive in a renderer-side cache (`useRef` / Zustand) and re-attached via `terminal.open(newDiv)`. React 19's `<Activity mode="hidden">` is the idiomatic primitive for "keep workspace tree mounted but paused." This is the same architecture v1.1.3 nominally targeted.

**Sidebar UI**: WorkspacesPanel.tsx already implements the V3 layout (added in v1.1.4) — colored dot, name, pane-count badge, close-X. The screenshot is achievable; what's missing is the **persistence reliability + scrollback preservation** that makes the experience feel real.

## What ships in v1.2.7

**Scope = the smallest set of changes that makes "open 5 workspaces, spawn 30 panes total across them, switch between, quit, relaunch, everything restored" actually work.** Approach 1 (ring-buffer replay) is cheaper than Approach 2 (keep xterm alive via `<Activity>`); we pick Approach 1 and queue Approach 2 for v1.3 if real-world latency is too high.

### 1. Ring-buffer replay on subscribe (THE keystone fix)

The registry already buffers every byte of PTY output. We just don't replay it when a renderer (re-)subscribes. Add a snapshot read.

- **Main**: new RPC `pty.snapshot(sessionId): { buffer: string }` that returns `record.buffer.read()` from the registry. Bounded at ~64KB to avoid IPC churn — drop the oldest if exceeded.
- **Renderer**: `Terminal.tsx` mount sequence becomes: (a) instantiate xterm, (b) call `rpc.pty.snapshot(sessionId)`, (c) `term.write(snapshot.buffer)`, (d) THEN `subscribePtyData(sessionId, cb)` for live stream. The renderer already deduplicates via xterm's internal write queue.
- **Net effect**: when user switches back to workspace A, each pane re-mounts with full scrollback. Looks identical to "never left."

### 2. Resume-launcher reliability — surface silent failures + extend extraction

The skip-when-no-`external_session_id` path is correct, but the user sees no signal when it happens. Two changes:

- **Extend scan window**: bump `externalSessionScanLineLimit` from 100 → 500 lines (registry.ts:105). Most CLIs print their session ID within the first 50 lines, but tools that interleave MCP-server-init output (which is what blew through the 100-line budget on v1.2.4–v1.2.5 boots) can push it past 100.
- **Surface resume failures**: `use-session-restore.ts:108` currently does `void rpc.panes.resume(item.workspace.id).catch(() => undefined)`. Replace with `.catch((err) => { toast.error(\`Couldn't resume \${item.workspace.name}: \${err.message}\`); console.warn(...) })`. Users see what failed instead of silently empty panes.
- **Log structured resume outcome**: `resume-launcher.ts` already returns `{ resumed: N, failedCount: N }` from line 240ish — pipe the failedCount into the toast so the user sees "Resumed 4 panes, 2 couldn't be restored."

### 3. Workspace switching is non-destructive (verify + document)

Per audit, this already works. We add a verification gate + 2 fresh tests so the next refactor doesn't break it:

- New vitest: `app/src/renderer/app/state.reducer.test.ts` — case "SET_ACTIVE_WORKSPACE_ID does NOT mutate sessions[]" + "WORKSPACE_OPEN doesn't dispatch any pty.* RPC."
- New playwright e2e: `tests/e2e/multi-workspace.spec.ts` — open 2 workspaces with 1 pane each, switch between them, assert pane count + PTY pid both unchanged. (Selector-stable test that survives sidebar redesigns.)

### 4. Sidebar UX polish to match V3 exactly

`WorkspacesPanel.tsx` is 95% there. The screenshot exposes 3 small gaps:

- **Chevron dropdown** at the top-right of the WORKSPACES header → opens a list of "persisted-but-closed" workspaces (rows from the `workspaces` table that aren't in `state.openWorkspaces`). Click → calls `rpc.workspaces.open(id)`. We have `state.persistedWorkspaces` (or equivalent) already populated; just need the dropdown UI.
- **Close-X visible on hover for every row** (not just active). v1.2.5 fix only showed it on active row. Make it show on `:hover` on every row, with `opacity-0` baseline + `group-hover:opacity-100`.
- **Active-row background**: confirm it matches `bg-accent/40` or whatever the V3 brand-blue tint is.

### 5. Docs + verification

- New `docs/04-design/multi-workspace-v1.2.7.md` — architecture doc covering the ring-buffer replay model, the resume-launcher reliability changes, and the v1.3 follow-up to migrate to React 19 `<Activity>` + xterm cache (Approach 2).
- `docs/08-bugs/BACKLOG.md` — close any open "session persistence" or "workspace switching" rows; add v1.3 row for "true xterm preservation via Activity."
- `CHANGELOG.md` — v1.2.7 entry.
- Manual smoke gate (build .app locally, install on a fresh kv, run the user's exact repro: 5 workspaces × 30 panes total × switch × quit × relaunch → everything restored with scrollback intact).

## Open risks

- **R-1.2.7-1**: Ring buffer size (~64KB) is small for long-running agents that have produced megabytes of output. Snapshot replay will show only the most recent ~1000 lines. Mitigation: acceptable for v1.2.7; v1.3 can bump to 256KB per pane + LRU eviction. Document the limit.
- **R-1.2.7-2**: First `pty.snapshot` call after a session has just been created may return empty if the agent hasn't printed anything yet. Mitigation: that's correct behaviour — there's nothing to replay yet. The live stream takes over immediately after.
- **R-1.2.7-3**: 500-line scan window for `external_session_id` extends the in-memory scan cost slightly (still bounded). No measurable impact at this scale.
- **R-1.2.7-4**: If two renderer subscribers both call `pty.snapshot` for the same session concurrently, they each get the full buffer — fine for our single-window use case. If we ever support multi-window in the future, switch to a "subscribe since seq" API.
- **R-1.2.7-5**: Resume failure toast may spam users if many sessions fail. Mitigation: aggregate into one toast per restart ("Resumed 7, 3 couldn't restore"), not per session.

## What's NOT in this scope

- **Approach 2 — React 19 `<Activity mode="hidden">` + xterm cache** for zero-latency switching with full state preservation. Queued for v1.3. The v1.2.7 ring-buffer replay delivers 95% of the perceived benefit at 20% of the LOC.
- **Per-workspace `BrowserManager` lifecycle** — out of scope; no evidence of coupling today.
- **Workspace rename / reorder UI** — defer until users ask.
- **Workspace export/import** — defer.
- **PTY snapshot for closed-but-not-forgotten sessions** (the 200ms grace window) — not useful; if pty is dead the replay shows the final output, then nothing.
- **Cross-workspace pane drag-and-drop** — V3 doesn't show this in the reference; defer.
- **The v1.2.6 stdio MCP switch** — independent of this work; v1.2.7 should rebase on whatever Codex ships for v1.2.6 (so `@playwright/mcp` is or isn't a dep, irrelevant to this plan).
