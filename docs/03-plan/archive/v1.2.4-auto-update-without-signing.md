# Plan — v1.2.4: Auto-update without code-signing certs

> Make the in-app auto-update channel actually deliver updates to opted-in users on **both** Windows (unsigned NSIS) and macOS (adhoc-signed DMG), without buying Apple Developer ID ($99/yr) or EV Authenticode certs ($300-700/yr). Approach 3 (custom update server / trust-anything endpoint) is parked as future work — not in this release.

## Context — why now

v1.2.3 just shipped, which restored the macOS channel manifest (`latest-mac.yml`) that had been missing from every Mac release since v1.1.4. The channel is now intact on both platforms. But the actual update delivery is broken:

- **Windows**: `electron-updater` defaults `verifyUpdateCodeSignature: true` and compares the downloaded EXE's Authenticode publisher CN against `latest.yml`'s `publisherName`. We ship unsigned, so the field is empty/missing and verification silently rejects the new EXE.
- **macOS**: `Squirrel.Mac` calls `SecStaticCodeCheckValidityWithErrors(newApp, kSecCSCheckAllArchitectures, designatedRequirement)` against the downloaded ZIP. Adhoc-signed apps have no `TeamIdentifier`, and the designated requirement reduces to a CD-hash-only check that's by definition unique per build. The new ZIP's hash never matches the installed app's CD requirement → `SQRLCodeSignatureErrorDidNotPass` → silent reject.

Net result today (post-v1.2.3): opted-in user clicks "Check for updates" → server returns "v1.2.4 available" → electron-updater downloads → signature verification fails → user sees nothing. Auto-update is theatre.

**Research conclusions** (from `electron-userland/electron-builder` source inspection + community workarounds 2024-2026):

- **Windows is solvable in one line**: set `build.win.verifyUpdateCodeSignature: false` in `electron-builder.yml`. `electron-updater` skips the publisher check; the OS itself doesn't gate unsigned EXE execution (the user has already clicked through SmartScreen once during install). Confirmed in `electron-builder/issues/6934` with side effects documented (loses EXE custom icon if `signAndEditExecutable: false` is also set — we don't set that, so no impact).
- **macOS Squirrel.Mac has no bypass flag**. `allowDowngrade`, `disableDifferentialDownload`, `disableWebInstaller`, `forceDevUpdateConfig` — none touch signature verification (verified by reading `MacUpdater.ts:107`, `AppUpdater.ts:91/359/585-586`). The third-party `zetaloop/Squirrel.Mac` fork patches `SecStaticCodeCheckValidityWithErrors` out of the framework, but author explicitly warns "do not use in production." Hard wall.
- **The community-blessed pattern for adhoc Mac**: set `autoDownload = false`, handle `update-available` ourselves, download the DMG with plain `https.get()`, save to `~/Downloads`, `shell.openPath()` it, let the user drag-to-Applications. Skips Squirrel.Mac entirely. Documented on `electron.build/auto-update.html` ("download the update manually... and call quitAndInstall() only when ready") and in every 2024-2026 thread on the topic.

This plan ships Approach 1 from the original options doc: **disable Windows signature verification + bypass macOS Squirrel.Mac with a manual DMG handoff**.

## Why / How / What — design framing

### 1. Windows — flip the flag

- **Why**: The Authenticode publisher check is the only thing blocking. Without a cert there's no publisher to match. Disabling the check is the sanctioned electron-builder workaround for adhoc/unsigned distribution.
- **How**: Add `verifyUpdateCodeSignature: false` to the `win:` block in `app/electron-builder.yml`. Confirm `latest.yml` is still uploaded by `release-windows.yml` (already done — v1.2.1 onwards).
- **What**: 1-line YAML change. No code path changes. `electron-updater`'s existing NSIS-runner flow handles download → silent run with `/S` flag.

### 2. macOS — manual DMG handoff (Squirrel.Mac bypass)

- **Why**: Squirrel.Mac is a hard wall and the third-party fork is unsafe for production. The community pattern is to never let electron-updater touch the disk on Mac.
- **How**:
  - Set `autoUpdater.autoDownload = false` (already set in `auto-update.ts:47`).
  - Wire `autoUpdater.on('update-available', ...)` handler:
    - On `darwin`: extract the DMG URL from `event.files[].url` (matches the asset attached to the GitHub Release), download it via the existing `NodeHttpClient.download()` at `src/main/core/skills/marketplace.ts:239-279`, write to `app.getPath('downloads')/SigmaLink-${version}.dmg`. Emit a new `app:update-mac-dmg-ready` event with `{ version, path }` to the renderer.
    - On `win32`: call `autoUpdater.downloadUpdate()` and let electron-updater proceed normally (the `verifyUpdateCodeSignature: false` flag from step 1 lets it complete).
  - Renderer (`UpdatesTab.tsx`) subscribes to `app:update-mac-dmg-ready` and shows a toast: "v1.2.4 downloaded → [Open installer]". Click → IPC call to `app.openUpdateDmg(path)` → main process runs `shell.openPath(path)` → DMG mounts in Finder. User drags SigmaLink.app over /Applications/ → quits old app → launches new one.
- **What**: 4 files touched (auto-update.ts, rpc-channels.ts, rpc-router.ts, UpdatesTab.tsx). Reuses the existing HTTP client at `marketplace.ts:195-317` — no new dependencies.

### 3. Progress + error UX

- **Why**: A 100-150 MB DMG over a slow link is a multi-minute download. The user clicked "Check for updates" 60 seconds ago; if we don't show progress, they'll think it stalled.
- **How**:
  - The existing `NodeHttpClient.download()` exposes an `onChunk(delta, total)` callback. Emit per-chunk progress events: `app:update-mac-dmg-progress` carrying `{ version, downloaded, total }`. Renderer aggregates into a progress bar in UpdatesTab.
  - On error (network drop, GitHub 5xx, disk full): emit `app:update-mac-dmg-error` with `{ version, error }`. Renderer shows a toast with a "Try again" button.
- **What**: 2 additional event channels in the allowlist. ~30 lines of progress/error wiring in `auto-update.ts`.

### 4. Renderer states + UpdatesTab UX

- **Why**: Today the UpdatesTab shows only "Last checked: …" + "Check for updates" button. Once auto-update actually delivers, the user needs visual states for *checking*, *available*, *downloading X%*, *ready to install*, *error*.
- **How**:
  - Add a 5-state machine in UpdatesTab via `useState`:
    1. `idle` — default
    2. `checking` — RPC in flight
    3. `available` (Windows only — electron-updater handles download itself)
    4. `downloading` (Mac only — driven by `update-mac-dmg-progress` events)
    5. `ready` (Mac only — `update-mac-dmg-ready` fired)
    6. `error`
  - Show distinct copy per state (e.g. "Downloading v1.2.4… 12.4 MB / 87.5 MB").
  - For Windows: rely on electron-updater's `update-downloaded` event → renderer shows "Restart to install" toast → user clicks → `autoUpdater.quitAndInstall()` triggers NSIS.
- **What**: ~80-line addition to UpdatesTab.tsx. No new dependencies (uses the existing `sonner` toast lib + the existing tab layout).

## Open risks

- **R-1.2.4-1**: macOS user has SigmaLink running while opening the new DMG → drag-to-Applications fails because the destination is in use. Mitigation: on receipt of `update-mac-dmg-ready`, the renderer toast says "Quit SigmaLink, then open the installer." Add a "Quit and install" button that calls `app.quit()` BEFORE `shell.openPath(dmg)`. The new app is launched fresh from /Applications after the user drops it.
- **R-1.2.4-2**: GitHub Releases asset URLs are CDN-redirected (the `*.dmg` URL in `event.files[]` may 302 to `objects.githubusercontent.com`). The existing `NodeHttpClient.download()` already handles up to 5 redirects (`marketplace.ts:265`), so this is covered — but verify in smoke.
- **R-1.2.4-3**: Download interrupted; partial DMG on disk. Mitigation: download to `*.dmg.part` first, rename to `*.dmg` only on completion. The existing client doesn't do this; add ~5 lines.
- **R-1.2.4-4**: User toggles opt-in OFF mid-download. Mitigation: abort the in-flight download (track an `AbortController` keyed by `version`); discard the partial file.
- **R-1.2.4-5**: Windows `verifyUpdateCodeSignature: false` carries documented MITM/tamper risk if the publish channel is ever compromised. Mitigation accepted: publish over HTTPS to GitHub Releases that we control via 2FA-protected push access.
- **R-1.2.4-6**: macOS DMG has `~/Downloads` mounted as quarantined volume? Test on Gatekeeper-enabled Sequoia/Tahoe — the user may still hit "Apple could not verify" on the new app. Mitigation: the in-DMG `README — Open SigmaLink.txt` we ship already documents the Terminal `xattr -cr` / System Settings escape hatch (v1.1.6). Re-link from the auto-update UI when needed.

## What's NOT in this scope

- **Approach 3 — custom update server** (e.g. `update.electronjs.org`-style endpoint) that bypasses signing entirely on both platforms. Parked as future work; revisit when we either outgrow GitHub Releases as a channel OR want truly silent auto-install on macOS.
- **Apple Developer ID + notarisation**. Still deferred indefinitely.
- **EV / OV Authenticode for Windows**. Same.
- **`zetaloop/Squirrel.Mac` patched framework**. Explicit non-production status; we don't ship third-party forks of Apple-derived crypto code.
- **Silent install on macOS** (drag-to-Applications without user interaction). Requires Squirrel.Mac signature bypass which doesn't exist legitimately.
- **In-app rollback** ("revert to v1.2.3"). Out of scope; users can manually re-install older releases from the GitHub Releases page.
- **Delta updates** (`differentialDownload`). The block-map delta path on Windows still works under `verifyUpdateCodeSignature: false`, but we don't actively test it — first time it's exercised will be v1.2.4 → v1.2.5. Note in release notes.
- **Linux AppImage auto-update**. Untested platform; out of scope.
