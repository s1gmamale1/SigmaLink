# v1.6.0 — Skills tab in right panel

> Status: **documented, not scheduled**. UX request from operator 2026-05-21. No hard dependency on W-4 (shell-first); could ship before or after.

## Problem

SigmaLink's right-rail icon strip currently surfaces Browser, IDE, and Sigma Assistant as discoverable panels. Skills (superpowers slash-commands, Ruflo MCP skills, custom skills installed into agents) are NOT surfaced anywhere in the UI — the only activation path is typing `/skill-name` in Sigma Assistant or relying on an agent to invoke them via the `Skill` tool.

This means:

- A new user cannot **discover** what skills are installed without browsing the filesystem.
- The mental model of "attach this skill to that pane" has no UI affordance — you'd type `/brainstorming` and hope Sigma picks it up.
- Skills are workspace-wide by implicit convention; there's no per-pane or per-agent scoping surface.

## Proposed model

A new "Skills" tab in the right-rail icon strip, placed next to Browser / IDE / Sigma Assistant. Click → right panel opens to a Skills view.

### Right-panel content

Two sections:

1. **Installed skills** (list view):
   - One row per skill: name, description (1 line), source (superpowers / Ruflo / custom).
   - Search / filter bar at top.
   - Click a row → expand inline to show full description, arguments, example usage.

2. **Active skills in this workspace** (separate area at bottom or top):
   - Shows which skills are currently bound to this workspace OR specific panes within it.
   - Empty by default; populated by drag-drop interactions.

### Drag-drop UX

- **Drag a skill row onto a pane** → that skill becomes available to the agent in that pane. Conceptually: future agent invocations in that pane can use the skill without explicit slash-command invocation (or with reduced friction — possibly via a per-pane skill-context preamble).
- **Drag a skill row onto the workspace tab/header** → skill becomes available workspace-wide (all panes inherit).
- **Drag a skill row onto a swarm card** → skill becomes available to every agent in that swarm.
- **Drop target highlight** — pane border / workspace card pulses while dragging to indicate valid drop zones.

### Removal UX

- Active-skill chip in the workspace/pane → click X to detach.
- Or drag the chip back into the Skills list to remove.

## Architectural questions (to resolve in plan-mode before implementation)

1. **Skill activation semantics** — does dragging onto a pane LITERALLY change agent behavior, or is it purely informational (UI nudge for the user to type `/skill-name`)? Both are valid. The former is more useful, the latter is much cheaper to ship.

2. **Persistence layer** — where do per-pane / per-workspace skill bindings live? New table `pane_skills (pane_id, skill_id, attached_at)` + `workspace_skills`? Or KV-store entries keyed by `pane:<id>:skills`? Need to integrate with cross-machine sync (skills installed on machine A but not B → graceful degradation).

3. **Skill registry source** — superpowers skills come from `~/.claude/plugins/cache/claude-plugins-official/superpowers/<version>/skills/`. Ruflo MCP skills come from the Ruflo daemon. Custom skills location TBD. Need a unified discovery API that the renderer can call to list available skills.

4. **Sigma Assistant interaction** — when a skill is attached to a pane, does Sigma Assistant know about it? Should Sigma's tool-calling change based on pane skill context? Or is this purely a UI affordance for the human operator (drag a skill to a pane → human knows to invoke it manually)?

## Phasing

If/when scheduled, recommend a 2-phase rollout to limit blast radius:

| Phase | Scope | Days |
|---|---|---|
| 1 | Skills tab + right panel + read-only list (no drag-drop, no persistence). Just a discovery surface — operator can browse installed skills, click for details, copy `/skill-name` to clipboard. Resolves the "how do I know what's installed" problem cheaply. | 2 |
| 2 | Drag-drop activation + persistence + per-pane / per-workspace binding. Decide architectural questions 1-4 above before starting phase 2. | 5-7 |

Phase 1 alone is shippable as a v1.5.7 or v1.6.0 polish packet. Phase 2 is a larger investment that should follow operator usage feedback on phase 1.

## Sequencing relative to W-4 (shell-first)

Independent. If W-4 ships first, skill bindings can flow into the shell-injected CLI commands as additional env / context. If W-5 ships first, the skill binding layer exists as a UI-only affordance until W-4 (or never) wires it into the actual spawn pipeline.

## Open questions for operator (resolve at plan-mode kickoff)

- Should the icon also surface in mobile / tablet layouts (if/when those exist)?
- Should there be a "favorites" or "recently used" sub-section?
- Is there a per-CLI-provider compatibility filter (some skills only make sense for Claude, others for any agent)?
- Does dragging a skill to a pane that's running a non-Claude provider (e.g. Codex) work? Or only Claude/Sigma-compatible providers?
