# Plan — v1.3.0: Per-pane session picker in Workspace Launcher

## Context

User wants two parallel UX flows for session resumption:

**Scenario A — Creating a new workspace.** After picking the directory + pane count + providers, the launcher shows a session picker step. For each pane's provider, the user sees existing sessions found in that cwd via disk-scan, and chooses either "New session" or "Resume <id>". Then launches.

**Scenario B — Re-opening a persisted workspace from the sidebar.** Same picker UX, but pre-populated with whatever was saved last time (the user's previous per-pane decisions). User confirms or changes per pane before continuing.

Today (v1.2.8) the launcher has 3 steps (Start → Layout → Agents) and the resume flow is silent and automatic — every saved row gets resumed with `--resume <id>` or `--continue` fallback. The user has zero agency. The sidebar's persisted-workspaces dropdown re-opens a workspace but skips straight to the Layout step with auto-resume behind the scenes. No session-picker UI exists.

Three parallel investigators closed today and aligned on the architecture:

- **`disk-scan-extension-audit`**: v1.2.8's `findLatestSessionId(providerId, cwd)` returns only the newest. Need a list variant — `listSessionsInCwd(providerId, cwd, opts?)` returning `Array<{ id, providerId, cwd, createdAt, updatedAt, title?, firstMessagePreview? }>`. Claude + Codex + Kimi + OpenCode are feasible in ~2-3 hours. Gemini disk layout is undocumented — defer with empty array placeholder.
- **`launcher-ux-audit`**: 3-step wizard (StartStep → LayoutStep → AgentsStep) at `app/src/renderer/features/workspace-launcher/Launcher.tsx`. Insert a new step **between AgentsStep and Launch** (NOT between Layout and Agents — sessions depend on which providers got assigned per pane). `Stepper.tsx` already abstracts step ordering; pure additive. No persistence today of per-pane provider config beyond the `agent_sessions` rows that resume-launcher walks.
- **`session-picker-ux-researcher`**: industry standard converges on the JetBrains two-tier pattern. Smart default (auto-pick newest in cwd for that provider) + per-pane "Change…" trigger that opens a Radix `Popover` with a shadcn `Command` (cmdk) full history list. Bulk-bar above the list ("Resume newest for all", "All new"). 0 clicks for the common path, full control for power users. Don't repeat Claude Code's "only 10 shown" mistake — show top 50 with a "Show all" escape.

This plan ships **Scenario A in full + the data layer for Scenario B**. The Scenario B "re-populate from last-time choices" pre-population is wired but is graceful — if no last-time data exists (fresh upgrade), Scenario B simply degrades to Scenario A.

## What ships in v1.3.0

A new optional Launcher step `SessionStep` that appears after AgentsStep. Smart default = newest session per (cwd, provider) tuple, override via per-pane chip → Popover with full Command list. Bulk bar with 3 actions. Wire the choices through to `executeLaunchPlan` so spawned panes either start fresh OR run with `--resume <id>`. Persist the per-pane resume plan so the next workspace-open pre-populates.

## Why / How / What

### 1. Extend disk-scanner to list all sessions

- **Why**: The picker needs a list, not the latest.
- **How**:
  - `app/src/main/core/pty/session-disk-scanner.ts` — add `listSessionsInCwd(providerId, cwd, opts?: { maxCount?: number; sinceMs?: number }): Promise<SessionListItem[]>`. Keep the existing `findLatestSessionId` for the v1.2.8 capture path — they coexist; the new function drops the 5-min mtime window and sorts DESC.
  - Per-provider extraction:
    - **Claude**: glob `~/.claude/projects/<slug>/*.jsonl` where slug = `cwd.replace(/\//g, '-')`; per file, read first line, parse as JSON (it's the session-init metadata blob), extract `created_at` + the first user message text (truncate to 80 chars). Return up to `maxCount` (default 50).
    - **Codex**: glob `~/.codex/sessions/**/rollout-*-<uuid>.jsonl` ; UUID from filename; `created_at` from filename ISO timestamp; first user message from JSONL line 1.
    - **Gemini**: return `[]` for v1.3.0. Format undocumented. File a backlog row for v1.3.1.
    - **Kimi**: list `~/.kimi/sessions/<sha1(cwd)>/<uuid>/state.json`; read JSON; extract `timestamp` + `model` + `first_user_message` if present.
    - **OpenCode**: shell `opencode session list --format json --max-count 50`, filter by `directory === cwd`, parse `{id, directory, updated, title}`.
  - Cap N at 50 to avoid Claude Code's documented "only 10 shown" complaint while keeping I/O bounded.
- **What**:
  - 1 new function + per-provider strategy in `session-disk-scanner.ts` (~150 LOC).
  - New zod schema + RPC + handler: `panes.listSessions(providerId, cwd, opts?)` at `app/src/main/core/rpc/schemas.ts`, `app/src/main/rpc-router.ts`, `app/src/shared/rpc-channels.ts`, `app/src/shared/router-shape.ts`.
  - Vitest cases mirroring the v1.2.8 `session-disk-scanner.test.ts` fixture pattern.

### 2. Add SessionStep to the Launcher wizard

- **Why**: The picker UI needs a slot. Insert it after Agents (so it knows per-pane provider) and before Launch.
- **How**:
  - New file `app/src/renderer/features/workspace-launcher/SessionStep.tsx` (~250 LOC):
    - Renders one row per pane (same iteration as AgentsStep).
    - Each row: provider color dot · provider name · pane index · session chip ("New session" by default OR "3h ago · feat/auth · 47 msgs") · "Change…" button opening a Radix `Popover`.
    - Popover content: shadcn `Command` (cmdk) with a search input, then a virtualized list of `SessionListItem` rows (timestamp + first-message-preview + short SHA + title). Hitting a row OR ↵ closes the popover and binds the chip.
    - Bulk bar at the top: "Resume newest for all" (sets all panes to their newest session for that provider/cwd) · "All new" (resets all to "New session") · "Custom…" (no-op, just signals user intent).
    - Smart default applied on step enter: for each pane, fetch `panes.listSessions(providerId, cwd)`, pre-select `sessions[0]` if it exists. If `sessions.length === 0`, leave at "New session".
  - Update `app/src/renderer/features/workspace-launcher/Stepper.tsx` — add `sessions` step between `agents` and the implicit launch action.
  - Update `app/src/renderer/features/workspace-launcher/Launcher.tsx`:
    - New state: `paneResumePlan: Record<paneIdx, { providerId, sessionId: string | null }>`.
    - In `launch()`, pass `paneResumePlan` through to the RPC.
  - Reuse existing utilities:
    - `workspace-color.ts` for provider colour dots.
    - shadcn `Popover` + `Command` + `Badge` primitives (all in repo).
    - sonner for any error toasts.
- **What**: 1 new component (~250 LOC), 1 Stepper edit, 1 Launcher state addition. No new dependencies.

### 3. Thread resume plan through to PTY spawn

- **Why**: The picker's output must actually drive the spawn, not just be UI theatre.
- **How**:
  - Extend `LaunchPlan` type at `app/src/shared/types.ts` (or wherever it's defined) with `paneResumePlan?: Array<{ paneIndex: number; sessionId: string | null }>`.
  - `app/src/main/core/workspaces/launcher.ts:executeLaunchPlan` — for each pane, if `paneResumePlan[idx].sessionId` is non-null, set `extraArgs` to the per-provider resume flag from `buildResumeArgs` (already exists in v1.2.8's `resume-launcher.ts`). Otherwise spawn fresh (current behaviour).
  - **Critical**: in the "resume" branch, also pre-stamp `agent_sessions.externalSessionId = sessionId` at insert time so v1.2.8's post-spawn capture path is a no-op.
- **What**: ~30 LOC across 2 files. No new schema.

### 4. Persist the resume plan for next-open pre-population (Scenario B)

- **Why**: When the user re-opens a persisted workspace from the sidebar dropdown, the picker should pre-populate with their last-time decisions, not re-derive from disk.
- **How**:
  - Don't add a new column. Reuse `agent_sessions` — the table ALREADY has `externalSessionId` per row, AND the row carries `workspaceId` + `providerId`. The "last time" snapshot is the most recent successfully-spawned rows per (workspace, paneIndex).
  - On Scenario B entry (sidebar dropdown → workspace open), the Launcher fetches `rpc.panes.lastResumePlan(workspaceId): Promise<Array<{ paneIndex, providerId, sessionId }>>` — new RPC reading the most recent `agent_sessions` rows where `workspaceId = ?` grouped by `paneIndex` and ordered by `startedAt` DESC.
  - SessionStep checks this pre-populated state. If present → use it as the initial chip values (override smart default). If absent → fall through to smart default.
  - Failure mode: if a pre-populated sessionId is no longer on disk (deleted, garbage-collected), the chip falls back to "Newest in cwd" silently with a console.warn. Don't surface as a user-blocking error.
- **What**: 1 new RPC (~25 LOC). 1 SessionStep useEffect to merge pre-populated values.

### 5. Route the sidebar dropdown through SessionStep

- **Why**: Today the sidebar's "open persisted workspace" advances directly to Layout step, skipping the picker.
- **How**:
  - `Launcher.tsx:101-105` `chooseExisting()` — after `workspaces.open()`, instead of advancing to Layout, jump directly to the SessionStep step **and** auto-fill Layout + Agents from the persisted workspace's last pane config (derive from the same `lastResumePlan` query).
  - The user lands on SessionStep with all chips pre-populated. They confirm or change, then click Launch.
  - **Edge case**: if the workspace has no persisted pane config (never launched before, just opened the folder), fall through to the full wizard flow starting at Layout. This is the "first-launch-as-workspace" case and matches Scenario A.
- **What**: 1 Launcher.tsx flow tweak (~20 LOC), 1 conditional skip for Layout+Agents when re-opening.

### 6. Bulk-bar actions

- **Why**: With up to 20 panes, per-pane manual selection is friction. The research strongly recommends bulk options.
- **How** (within SessionStep):
  - **"Resume newest for all"** — for each pane, if `listSessions(providerId, cwd)[0]` exists, bind chip to it. Otherwise leave at New.
  - **"All new"** — reset every chip to "New session", clearing `paneResumePlan` to `{ sessionId: null }` for every pane.
  - **"Reset to suggested"** — re-runs the smart default logic (overwrites any user changes).
- **What**: 3 button onClick handlers in SessionStep (~30 LOC total).

## Open risks

- **R-1.3.0-1**: Reading the first line of N JSONL files for the picker is N file opens. At max 50 per provider, ~50 stat+open is acceptable (~25ms in profiling). If a user has thousands of historical Claude sessions, slow disks could surface. Mitigation: lazy-load on Popover open per pane, NOT eagerly on step entry. Smart default only fetches `listSessions` synchronously for the auto-pick.
- **R-1.3.0-2**: Gemini disk layout is undocumented. The picker shows an empty list for Gemini panes — acceptable for v1.3.0. File backlog row "Gemini session disk-scan support" for v1.3.1.
- **R-1.3.0-3**: Pre-populated session IDs may be stale (file deleted, GC'd). Mitigation: validate against disk-scan results on SessionStep mount; silently fall back to newest if invalid; console.warn for debug visibility.
- **R-1.3.0-4**: Scenario B "skip Layout + Agents" optimisation may surprise users who wanted to change pane count when re-opening. Mitigation: add a small "Reconfigure layout…" link at the top of SessionStep that navigates back to Layout step.
- **R-1.3.0-5**: 20-pane × 50-session-each = 1000 disk entries to enumerate on Bulk "Resume newest for all" — but each pane only needs its top 1, so the bulk button reuses the cached smart-default values. No re-fetch.

## What's NOT in this scope

- **Gemini session disk-scan support** — deferred to v1.3.1.
- **OpenCode SQLite direct read** — keep using `opencode session list --format json` subprocess.
- **Cross-machine session sync** — deferred (in WISHLIST.md).
- **Session deletion / rename from picker** — view-only. Deferred.
- **Session metadata search across non-cwd paths** — picker scoped strictly to current cwd.
- **Importing sessions from outside SigmaLink-managed cwd** — out of scope.
- **W-2 (Sigma Assistant orchestrator) + W-3 (Auto-bind Ruflo MCP)** — separate releases; v1.3.0 ships W-1 only.
