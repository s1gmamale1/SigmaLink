# Plan — v1.2.5: Post-install regressions from v1.2.4

> User installed v1.2.4 macOS DMG via the curl-bash one-liner and reported 6 visible symptoms: empty Workspaces sidebar, "+ Pane" button no-op, 6th pane (Kimi) crashing with `ioctl(2) failed, EBADF — pty.resize`, all CLI panes showing "MCP client for `browser` failed to start: HTTP request failed: error sending request for url (http://127.0.0.1:58360/mcp)", Ruflo readiness pill red, Focus button does nothing. v1.2.5 fixes the actual regressions and clarifies the false alarms.

## Context — what we found

4 parallel investigators (workspaces / mcp / pty-ebadf / focus-ruflo) confirmed the symptoms split into **two real regressions + four downstream / pre-existing issues**:

### Real regressions

1. **Playwright browser MCP supervisor never starts in the packaged app.** The URL `127.0.0.1:58360/mcp` is NOT SigmaLink's internal `mcp-host-bridge` (that uses Unix domain sockets at `/tmp/sigma-host-<pid>-<uuid>.sock` — verified in `app/src/main/core/assistant/mcp-host-bridge.ts:84-89, 108-131`). It's the **Playwright browser MCP supervisor** (`app/src/main/core/browser/playwright-supervisor.ts:95-120`) which spawns `@playwright/mcp` as a child process on a dynamically-allocated port. Two failure modes (both real):
   - `@playwright/mcp` is in `devDependencies` (line 91 of `package.json`), not `dependencies` — electron-builder excludes devDeps from the packaged DMG. `createRequire(import.meta.url).resolve('@playwright/mcp/package.json')` throws.
   - Fallback at `playwright-supervisor.ts:168` tries `npx @playwright/mcp@0.0.75`. On macOS DMG-launched Electron, `process.env.PATH` is the OS-minimal `/usr/bin:/bin:/usr/sbin:/sbin` (no `/opt/homebrew/bin`, no Node bin dirs). `npx` is at `/opt/homebrew/bin/npx` for arm64 Homebrew users → ENOENT, supervisor's spawn callback writes `entry.lastError` but **doesn't broadcast or log**, so the failure is invisible. Supervisor's readiness probe at line 119 has a 4s timeout that catches and discards the failure: `await waitForPortListening(port, 4000).catch(() => undefined);`

2. **PTY `proc.resize()` is unprotected and crashes on dead handles during the 200ms graceful-exit window.** `app/src/main/core/pty/local-pty.ts:306-307` does:
   ```ts
   resize: (cols, rows) => proc.resize(...),
   ```
   No try-catch. Compare to `kill` at line 308-313 which already has the try-catch idiom. The PtyRegistry at `registry.ts:218-220` forwards the call without checking the `alive` flag. When a pane exits quickly (e.g. Kimi spawn → ENOENT → exit within ms), the renderer's `ResizeObserver` at `Terminal.tsx:192-204` fires inside the 200ms grace window and calls `pty.resize` → node-pty throws `Error: ioctl(2) failed, EBADF` → IPC wraps to `{ok: false, error}` → toast appears.

### Downstream / pre-existing

3. **`+ Pane` button silently disabled.** `CommandRoom.tsx:198-225` disables the dropdown trigger when `!activeSwarm || activeSwarm.status !== 'running'`. With workspaces empty (regression 4 below) or swarm-list empty, the button locks with no user-visible reason. **Fix is UX**: add a tooltip with the disabled-reason ("No active swarm — open or create a workspace first").

4. **Workspaces sidebar appears empty.** Workspaces investigator flagged a possible auto-update boot race in `auto-update.ts:18-39` (`maybeCheckOnBoot` calls `kvGet` synchronously before DB warmup). But on inspection, `kvGet` has its own try-catch returning `null` on failure, so it shouldn't poison init. **More likely cause**: this is a fresh install with no prior `~/Library/Application Support/SigmaLink/sigmalink.db` data — the user just installed v1.2.4 and the empty sidebar is the genuine empty state. The user's screenshot DOES show "Workspace 1 / aisigma — app" in the top bar, so a workspace IS active and 6 panes are running — confirming the DB is functional. The "empty sidebar" complaint is likely about the V3 BridgeMind-style WorkspacesPanel not rendering its full row (could be a small CSS regression OR the workspace just wasn't named so the row is invisible). Needs a quick UI sanity check, not a deep fix.

5. **Ruflo readiness pill red.** `RufloReadinessPill.tsx:73-89, 120-137` turns red when `ruflo.verifyForWorkspace()` returns `claude && codex && gemini` all false OR `skills.verifyForWorkspace()` returns `verified===0`. Both RPCs walk through the broken Playwright MCP chain (the Ruflo verify probe tries to invoke each CLI's `/mcp list` which fails for the `browser` server). **Fix is downstream of regression 1** — once Playwright MCP works, the pill goes green automatically.

6. **Focus button does nothing.** Not a regression — never implemented. `PaneHeader.tsx:109` calls `ctx.activate()` → `SET_ACTIVE_SESSION` which only sets the focus ring, not fullscreen. No `focusedPaneId` state or fullscreen CSS exists. **Fix options**:
   - (a) Implement actual fullscreen (hide siblings, expand active pane to fill grid). ~80 LOC.
   - (b) Relabel the icon (`Maximize2` → `Target`) and rename to "Pin focus" so users don't expect fullscreen.
   - **Decision**: (b) for v1.2.5 (small, honest), (a) for v1.3 with the real Split + Minimise implementations.

### What's NOT broken (but I want to make sure)

- **DB / userData path**: `app.getPath('userData')` resolves to `~/Library/Application Support/SigmaLink/sigmalink.db`. `productName` and `appId` unchanged across v1.2.0-v1.2.4. No data-loss bug.
- **Adhoc codesign + spawn-helper chmod**: v1.2.4 includes the chmod fix from `e193943`. NativeRebuildModal does NOT pop in the user's screenshots, confirming the chmod fix worked.
- **MCP host bridge** (Unix socket / Sigma Assistant tools): different subsystem; unrelated to the browser-MCP HTTP failure.
- **Provider registry**: 5 CLIs only (Claude, Codex, Gemini, Kimi, OpenCode). Working.

## Why / How / What — design framing

### 1. Playwright MCP packaging + PATH augmentation (the keystone fix)

- **Why**: Single root cause for 3 visible symptoms (browser MCP errors, Ruflo pill red, possibly the "MCP issues detected" Gemini banner). User can't actually use the CLI agents for any browser-needing task today.
- **How**:
  1. **Move `@playwright/mcp` from `devDependencies` to `dependencies`** in `app/package.json`. electron-builder will then include it in the packaged DMG. `pnpm install` to refresh lockfile.
  2. **Augment `process.env.PATH` at app boot on macOS** to include `/opt/homebrew/bin`, `/usr/local/bin`, `~/.volta/bin`, `~/.nvm/versions/node/*/bin`. We already do this for shell PATH bootstrap (`electron/main.ts:bootstrapShellPath`), but the supervisor.ts spawn inherits `process.env` which is the OS-minimal subset on Finder-launched Electron. Add a `bootstrapNodeToolPath()` helper that runs at boot and prepends the standard Node tool dirs.
  3. **Surface supervisor errors**: in `playwright-supervisor.ts`, when spawn fails, broadcast `app:browser-mcp-failed` event with the error message so the Ruflo pill OR a toast can tell the user "Browser MCP didn't start — install @playwright/mcp globally OR run from terminal." Today it's swallowed.
- **What**: 1 package.json move + 1 new boot helper (~25 LOC) + 1 error broadcast (~10 LOC in supervisor).

### 2. PTY resize crash hardening

- **Why**: EBADF is one symptom of a class of bugs where the renderer races the PTY exit. Fix is small and tightens a latent race.
- **How**:
  1. `app/src/main/core/pty/local-pty.ts:306-307` — wrap `proc.resize(...)` in try/catch like `kill` at 308-313. Silently swallow EBADF; throwing into IPC just causes the toast spam.
  2. `app/src/main/core/pty/registry.ts:218-220` — add `if (!session?.alive) return;` guard above the resize call. Mirrors the same idiom in `kill()`.
  3. `app/src/renderer/features/command-room/Terminal.tsx:192-204` — subscribe to `pty:exit` event and call `resizeObserver.disconnect()` when the pane's PTY exits. Defensive cleanup so the observer doesn't keep firing for a dead session.
- **What**: ~10 LOC across 3 files. Add a vitest case in `registry.test.ts` that asserts `registry.resize(deadSessionId, …)` is a no-op.

### 3. Add Pane disabled-state tooltip

- **Why**: User clicks a dead button with no feedback. Worst kind of "broken."
- **How**: `app/src/renderer/features/command-room/CommandRoom.tsx:198-225` — wrap the disabled DropdownMenuTrigger in a Radix `Tooltip` that explains why:
  - `!activeSwarm` → "Open or create a workspace first"
  - `activeSwarm.status !== 'running'` → "Swarm is paused — resume to add panes"
  - `activeSwarm.agents.length >= 20` → "Max 20 agents per swarm (current: 20)"
- **What**: 1 Tooltip wrapper + 1 helper function returning the disabled-reason string. ~20 LOC.

### 4. Workspaces sidebar empty-state check

- **Why**: User reports empty sidebar despite an active workspace. Likely a CSS or render-empty-list bug in `WorkspacesPanel`, OR the workspace row truly is empty (no name) and the row is invisible.
- **How**:
  1. Read `app/src/renderer/features/sidebar/WorkspacesPanel.tsx` end-to-end. Confirm the empty-state UI exists ("No workspaces yet. Click + to add one.").
  2. Confirm the active-workspace row renders with a fallback name when `workspace.name` is empty (e.g. "Untitled workspace" + short path).
  3. If both check out and the panel still doesn't render, add a quick visual smoke test (`pnpm exec playwright test tests/e2e/sidebar.spec.ts`).
- **What**: Read-only first; only edit if the empty-state path is genuinely missing.

### 5. Focus button honest relabel

- **Why**: User expects fullscreen because the icon (`Maximize2`) implies it. Today it only pins the focus ring.
- **How**: `app/src/renderer/features/command-room/PaneHeader.tsx` — swap the icon to a `Target` or `Crosshair` lucide icon and update the tooltip label to "Pin focus (Cmd+Alt+N)". Add a v1.3 BACKLOG row for "implement true pane fullscreen."
- **What**: 1-line icon change + 1 tooltip update. ~3 LOC.

### 6. Verification — agents launch the actual app

Per user directive: fix agents must **launch the packaged app, run a Playwright smoke, and capture screenshots** before declaring done. The existing `tests/e2e/smoke.spec.ts` covers the boot flow but has stale v1.1.4 selectors per BACKLOG.md. Each fix agent's "definition of done" includes:
1. The targeted symptom reproduced before the fix (via screenshot of the unfixed state).
2. The symptom verified-fixed via screenshot after the fix.
3. `pnpm run build && node scripts/build-electron.cjs && pnpm exec electron-builder --mac --dir --publish never` produces a working `.app` bundle in `app/release/mac-arm64/`.
4. Launch the .app via `open app/release/mac-arm64/SigmaLink.app`, wait 5s for Playwright supervisor to start, screenshot.

## Open risks

- **R-1.2.5-1**: `@playwright/mcp` is large (~50MB with browser binaries downloaded on first run). DMG size grows from ~140MB → ~190MB. Acceptable; modern.
- **R-1.2.5-2**: PATH bootstrap might still miss user's specific Node install location. Mitigation: surface the error broadcast (Step 1's error event) so users know what failed and can fix manually.
- **R-1.2.5-3**: Disabling EBADF errors silently could mask a different real bug. Mitigation: still log to console with `[pty]` tag so the issue is debuggable; only suppress the IPC toast.
- **R-1.2.5-4**: Focus button relabel might confuse users who memorised the v1.1.4 "expand" semantics. Mitigation: BACKLOG row documents the v1.3 promise to actually implement fullscreen.

## What's NOT in this scope

- True pane fullscreen for Focus button (v1.3).
- Pane Split + Minimise functionality (v1.3).
- Kimi npm package name verification + auto-install hint (deferred; current workaround: users install Kimi CLI manually before opening a Kimi pane).
- Playwright e2e selector refresh from v1.1.4 (BACKLOG.md "Playwright e2e refresh" row — separate scope).
- Gemini "MCP issues detected. Run /mcp list for status." banner — that's Gemini's own CLI surfacing config issues; cascades from Playwright MCP being down.
- Gemini "Update available 0.41.2 → 0.42.0" prompt — Gemini's own CLI; not our concern.
- Auto-update boot order race (workspaces-investigator hypothesis 1+2) — unlikely root cause based on `kvGet` already having try/catch; revisit if symptoms persist after Step 1.
- Notifications system.
- Apple Developer ID / EV cert.
