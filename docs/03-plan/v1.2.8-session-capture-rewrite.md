# Plan — v1.2.8: Session capture rewrite (pre-assign UUID + disk-scan + --continue fallback)

> User reported on Windows: spawned 4 panes in a workspace, quit before interacting, relaunched → toast "Could not resume 4 panes: missing external_session_id." Root cause: the v1.2.7 toast surfaces a real bug — none of the 4 panes had their session ID captured because (a) all CLIs except Claude have no extraction implemented, and (b) Claude prints its session ID only AFTER the MCP approval prompt is dismissed. Investigator confirmed. CLI research surfaced a cleaner approach: every provider supports `--continue` (resume-latest) and two providers (Claude, Gemini) support `--session-id <uuid>` for pre-assignment. Replace the fragile stdout extractor with a hybrid that never needs to scrape banner output.

## Context — why now

v1.2.7 shipped resume-failure visibility — the toast works as intended. But the underlying capture is broken for 4 of 5 providers (the stdout extractor at `session-id-extractor.ts:51-105` only handles `claude`, returns `null` for `codex|gemini|kimi|opencode`). Even for Claude, the extractor's 500-line scan can't see past the MCP server approval prompt that blocks startup output until the user clicks through. The user's Windows screenshot is a perfect repro.

The user's proposal — actively interrogate the CLI instead of scraping — is correct. A 1-agent research pass confirmed every provider has a path:

| Provider | Pre-assign UUID at spawn? | Resume by ID? | Resume latest in cwd? | Disk path for capture |
|---|---|---|---|---|
| claude | **Yes** — `--session-id <uuid>` | `--resume <uuid>` / `-r` | `-c` / `--continue` | `~/.claude/projects/<cwd-slug>/<uuid>.jsonl` |
| codex | No flag — spawn then disk-scan | `codex resume <uuid>` | `codex resume --last` | `~/.codex/sessions/YYYY/MM/DD/rollout-*-<uuid>.jsonl` |
| gemini | **Yes** — `--session-id <uuid>` | `--resume <uuid>` | `--resume latest` | `~/.gemini/history/<project>/` |
| kimi | No flag — spawn then disk-scan | `--session <uuid>` / `-S` | `-C` / `--continue` | `~/.kimi/sessions/<project-hash>/<uuid>/` |
| opencode | No flag — spawn then disk-scan | `--session <id>` / `-s` | `-c` / `--continue` | SQLite `~/.local/share/opencode/opencode.db` + `opencode session list --format json` |

**Single most important finding**: all five providers support a `--continue`-style "resume latest in cwd" flag. That's our universal fallback.

**Correction**: Kimi CLI is **PyPI not npm**. `pip install kimi-cli` (or `uvx kimi`), NOT `npm i -g`. Our `providers.ts` install hint and onboarding copy are wrong.

## What v1.2.8 ships

A unified hybrid session strategy across all 5 providers:

1. **At pane spawn** —
   - For **claude + gemini**: SigmaLink generates a UUID locally (`crypto.randomUUID()`) BEFORE spawn, passes it via `--session-id <uuid>`, and writes the DB row with that `external_session_id` immediately. **Zero extraction. Zero race.**
   - For **codex + kimi + opencode**: spawn normally. Two seconds after spawn, async disk-scan the provider's deterministic session directory (filename or dirname = UUID), pick newest mtime scoped to the pane's worktree path, write to DB row.
2. **At resume** —
   - Try captured `external_session_id` via provider-specific resume flag.
   - If missing OR if the resume RPC fails: fall back to universal `--continue`/`--last`/`--resume latest`. Don't error — try the universal flag silently and report only if THAT also fails.
3. **Delete the stdout extractor**. The 250-LOC `session-id-extractor.ts` + the in-registry scan loop go away. Net -300 LOC.
4. **Add "Re-spawn fresh" button** to the resume-failure toast as the ultimate fallback (worktrees + branches are preserved on disk; user just gets fresh PTYs in them).
5. **Fix Kimi install hint** — `providers.ts` `installHint: 'pip install kimi-cli'` (or pointer at `https://github.com/MoonshotAI/kimi-cli`).

## Work breakdown

### Step 1 — Spawn-time capture refactor (M, ~2 hr)

**Main**:
- `app/src/main/core/pty/launcher.ts` (or wherever providers are spawned) — before passing args to PTY, check provider id:
  - `claude` or `gemini` → generate UUID, prepend `--session-id <uuid>` to args, store `preassignedSessionId = uuid` on the spawn config.
  - others → spawn args unchanged.
- `app/src/main/core/workspaces/launcher.ts` — when inserting the `agent_sessions` row, use `preassignedSessionId` directly as `external_session_id` for claude/gemini. For others, leave NULL initially.
- New `app/src/main/core/pty/session-disk-scanner.ts` — exports `findLatestSessionId(providerId, cwd): Promise<string | null>` that:
  - For `codex`: globs `~/.codex/sessions/**/rollout-*.jsonl` (only files mtime within last 5 min), filters by cwd ancestry (codex stores cwd inside the JSONL — first-line metadata; or just take newest by mtime as v1 simplification), extracts UUID from filename.
  - For `kimi`: lists subdirs of `~/.kimi/sessions/<project-hash>/`, sorted by mtime, returns newest UUID. Project-hash = SHA1(cwd) per kimi convention.
  - For `opencode`: spawns one-shot `opencode session list --format json --max-count 10`, parses JSON, filters by `directory === cwd`, returns newest by `updated`.
- New trigger in `app/src/main/core/pty/registry.ts` `create()` → after 2s post-spawn, fire `findLatestSessionId(providerId, cwd)` async, on resolve `db.update(agent_sessions).set({external_session_id: id})`. No blocking. Bounded retry: try at +2s, +5s, +15s.

**Tests**:
- `session-disk-scanner.test.ts` (NEW) — golden cases per provider against synthetic disk fixtures.

### Step 2 — Resume strategy rewrite (S, ~1 hr)

**Main**:
- `app/src/main/core/pty/resume-launcher.ts`:
  - For each pane to resume, build args per provider:
    - claude with id → `['--resume', sessionId]`
    - claude without id → `['--continue']`
    - codex with id → `['resume', sessionId]`
    - codex without id → `['resume', '--last']`
    - gemini with id → `['--resume', sessionId]`
    - gemini without id → `['--resume', 'latest']`
    - kimi with id → `['--session', sessionId]`
    - kimi without id → `['--continue']`
    - opencode with id → `['--session', sessionId]`
    - opencode without id → `['--continue']`
  - Treat "missing external_session_id" as a SUCCESS path (it just routes to `--continue`), not a failure.
  - Only report a failure when the universal fallback ALSO fails (the PTY spawn itself errors).

## Open risks

- **R-1.2.8-1**: `--session-id <uuid>` flag behavior for claude/gemini may differ across CLI versions. Test against the user's installed versions; fall back to disk-scan if pre-assign fails.
- **R-1.2.8-2**: Disk-scan for codex/kimi/opencode picks "newest mtime in cwd" — if the user runs the same provider outside SigmaLink in the same cwd, we may capture the wrong session. Mitigation: scope scan to files created within last 5 min of spawn; cross-reference with provider's project-hash if available.
- **R-1.2.8-3**: `--continue` semantics differ per provider on "what's the most recent." Claude scopes by cwd; gemini scopes globally with `latest` token; codex scopes by cwd + `--all`. Document the scoping clearly.
- **R-1.2.8-4**: Kimi PyPI install requires Python — users without it can't install. Acceptable; document in install hint. Kimi already has a `kimi.cmd` shim on Windows after `pip install`.
- **R-1.2.8-5**: Disk-scan adds I/O on every pane spawn. Negligible (~1 stat call), but document.

## What's NOT in this scope

- **OpenCode SQLite direct read** (instead of `opencode session list --format json` subprocess). Defer — subprocess works, SQLite read needs the file schema reverse-engineered.
- **`/status` slash command interrogation** — the user's Approach A specifically; turns out it's not needed because `--session-id` (claude/gemini) + disk-scan (others) is cleaner and doesn't require spawning a second process.
- **Provider auto-install** ("kimi not found → run pip install for me?"). Defer.
- **Cross-machine session sync**. Defer.
- **The two v1.2.7 R-1.2.7-* polishes** (Terminal.tsx mount race, toast aggregation) — fold the toast aggregation into Step 4 since we're rewriting that surface; defer the Terminal.tsx race to v1.3.
