# v1.6.0 — Shell-first pane architecture

> Status: **documented, not scheduled**. Trigger: v1.5.6 user diagnostic data confirms PTY-direct-CLI model is structurally brittle, OR after one more recurrence of the empty-pane class of regressions.

## Problem

Today (v1.5.6 and earlier), SigmaLink spawns a PTY whose direct child is the CLI binary (`claude`, `codex`, `opencode`, etc.). The PTY process IS the CLI. When the CLI exits — Ctrl+D, `/quit`, crash, or fast-fail during startup — the PTY closes and the pane is dead. There is no shell to fall back to.

Consequences:

- **Empty-pane class of regressions**: any startup-time failure of the CLI (ENOENT, missing PATH, version-incompatible flag) results in a closed PTY with no error visible to the user. v1.5.6 mitigates this by widening the ring-buffer grace window to 3 s so the renderer can read the binary's stderr before it disappears, but the underlying fragility remains.
- **Resume complexity**: SigmaLink tracks `external_session_id` in DB, builds per-provider resume args (`--resume <uuid>`, `--continue`, disk-scan JSONL bridges), and injects them at spawn. ~150 code references across the codebase serve this single concern.
- **One-shot panes**: the user cannot run a second CLI in the same pane. If you `/quit` Claude in pane 3, the pane is gone — you cannot then type `codex` to switch.
- **No "extra terminal" affordance**: there is no easy way to drop into a shell for ad-hoc commands in the pane's worktree.

## Proposed model (SigmaSpace-style)

The PTY parent process becomes the user's interactive shell (`/bin/zsh` on macOS, `/bin/bash` Linux, `pwsh` / `cmd` Windows). The CLI becomes a grandchild process launched by the shell.

```
node-pty
  └─ /bin/zsh  (PTY process)
       └─ claude (CLI, started by shell after we write `claude\n` into PTY master)
```

On launch:

1. Spawn shell as the PTY child. Shell's startup files (`.zshrc`, `.bashrc`) execute normally — `PATH`, aliases, env all match what the user gets from a regular Terminal.
2. After shell prompt appears, programmatically write `<provider-cmd><Enter>` into the PTY master. Shell reads it, executes the CLI as a foreground child.
3. CLI prints banner, accepts input as normal.

On CLI exit:

- Ctrl+D / `/quit` / crash → CLI process exits. Control returns to shell prompt. Pane stays alive.
- User can now type any shell command (`ls`, `git diff`, `claude --resume <id>`, `codex`, etc.).
- Cmd+T inside a pane could open a sub-tab running a fresh shell (independent of the original CLI session).

On workspace resume:

- Spawn shell, write `<provider-cmd> --resume<Enter>` (or equivalent). Each CLI handles its own resume; SigmaLink no longer mediates session ids.
- DB persistence reduces to: workspace id, pane index, provider id (just "which CLI was assigned to this pane"). No `external_session_id`, no JSONL bridges, no disk-scans.

## Trade-offs

**Pros**:
- Durability: pane survives CLI exits.
- Composability: multiple CLIs sequentially in the same pane.
- Simpler resume: each CLI manages its own state via its native resume flag.
- Better error visibility: shell prints `claude: command not found` instead of the pane silently dying.
- Removes ~150 references to `external_session_id`, `claudeSlugForCwd`, `prepareClaudeResume`, disk-scan logic, capture pipelines, etc.
- v1.5.5 worktree-suffix UX feature survives intact (the shell session still gets the session-id-correlated worktree).

**Cons**:
- Migration cost: ~150 references touched. ~2-3 week packet, not a hotfix.
- Notification UX changes: "agent done" detection currently uses `isResume + exit code` from the PTY. Shell-mode loses that signal — would need to parse shell output, watch process trees, or use shell hooks (`PROMPT_COMMAND`, `precmd_functions`, etc.) to detect CLI-process-exited-and-returned-to-prompt.
- Cross-machine sync currently snapshots `agent_sessions.external_session_id`. Schema needs to drop that field (or mark deprecated) — affects migration 0019 wire format.
- Sigma assistant dispatch ("/dispatch a Claude pane with this initial prompt") becomes `echo "<prompt>" | claude` or `claude --print "<prompt>"`. Needs reauthoring.
- Auto-launch timing: shell prompt may not be ready when we write `<provider-cmd>\n`. Need to either wait for a prompt regex or use `expect`-style readiness probe.

## Decision triggers

Ship this packet if any of:

1. v1.5.6 unmasked-error data confirms binaries are dying for environmental reasons that the shell-first model would naturally surface (e.g. "claude not in PATH" — shell would show this; current model swallows it).
2. A second occurrence of the empty-pane class of regressions ships in any v1.5.x — proving the PTY-direct-CLI model has more latent fragility than v1.5.6's grace window covers.
3. User explicitly requests the SigmaSpace-style multi-CLI-per-pane composability for workflow reasons.

Without one of those triggers, the v1.5.6 grace-window fix may be sufficient and shell-first is over-engineering.

## Incremental rollout strategy (W-4 started 2026-05-21)

This is a ~14-day core-PTY change that MUST NOT leave the app in a broken intermediate state. Strategy: a feature flag makes every phase independently shippable.

- **Flag**: KV setting `pty.spawnMode` ∈ `'direct'` (DEFAULT — today's PTY-direct-CLI behavior, zero regression) | `'shell-first'` (new). Settings → an "experimental" toggle.
- **Invariant**: with the flag at `'direct'`, behavior is byte-for-byte identical to pre-W-4. Every phase ships with the flag defaulting `'direct'`, so each release is safe even mid-migration. The default flips to `'shell-first'` ONLY after Phase 7 (full validation + dogfood).

### Phase 1 (this packet — v1.10.0, flag default 'direct')

Change ONLY the launch mechanism in `spawnLocalPty` (`app/src/main/core/pty/local-pty.ts:199`):
- `'direct'` mode: spawn `input.command` + `input.args` directly (current `platformAwareSpawnArgs` path) — UNCHANGED.
- `'shell-first'` mode (when `input.command` is a CLI, not already a shell): spawn `defaultShell()` as the PTY process, then after a prompt-ready signal (first `onData` chunk from the shell, with a ~250 ms fallback timer + double-inject guard), write the safely-shell-quoted composed command line `<command> <args>\n` into the PTY master. The shell launches the CLI as its child.
- The resume-arg machinery (`external_session_id`, `buildResumeArgs`, per-provider bridges) is UNTOUCHED in Phase 1 — `command + args` already include resume args; shell-first just writes that composed line to the shell instead of spawning it directly. Resume simplification is Phase 2.
- Consequence (intended): in shell-first mode, when the CLI exits the shell stays alive (pane durability — the whole point). Exit-detection rework is Phase 4; for Phase 1 the shell-prompt-after-exit is the desired behavior.

Phase 1 is well-bounded: it's a launch-mechanism branch in one function + a flag + a Settings toggle + tests for both modes. Default-off ⇒ no regression; opt-in ⇒ the new durable model for early dogfooding.

### Phase 2 (this packet — sentinel-based CLI-exit detection, flag default 'direct')

**Re-ordering note**: the original sequencing table above listed Phases 2-3 as
resume simplification and schema drop of `external_session_id`. Those phases are
**re-ordered to AFTER Phase 7 (the default-flip)** because direct mode still uses
`external_session_id` pre-flip and removing it mid-migration would break direct-mode
resume. The pre-flip additive phases are now:

| Phase | Scope |
|---|---|
| **2 (this packet)** | Exit-detection: sentinel injection + registry detection → cli-exited signal + notification |
| 3 | Dispatch rewrite: Sigma/Jorvis `echo … | cli` / `cli --print` pattern |
| 4 | Cmd+T sub-tab affordance + UI work |
| 5 | win32 shell-first quoting |
| 6 | Tests + Opus reviewer cycle + dogfood |
| 7 | Flip default to `'shell-first'` |
| 8 (post-flip) | Resume simplification: replace per-provider resume-arg builders |
| 9 (post-flip) | Schema drop of `external_session_id` (migration 0020) + CRDT replication update |

Phase 2 implementation details:
- **Sentinel constants**: `app/src/main/core/pty/sentinel.ts` — `SENTINEL_PREFIX = '__SIGMALINK_CLI_EXIT_'`, `SENTINEL_SUFFIX = '__'`. Shell prints `\n__SIGMALINK_CLI_EXIT_<code>__\n` after CLI exits.
- **Sentinel format**: `<command> <args>; printf '\n%s%d%s\n' '__SIGMALINK_CLI_EXIT_' "$?" '__'\n`
- **Detection regex**: `/(?:^|\r?\n)__SIGMALINK_CLI_EXIT_(\d+)__\r?(?:\n|$)/g`
- **Status representation choice**: a NEW `onCliExited` callback in `PtyRegistryOptions` (separate from `onPaneEvent`). Rationale: `sigma_pane_events.kind` has a SQLite enum constraint — adding `'cli-exited'` would require a migration (invasive). A separate sink is additive, zero-migration, and direct-mode consumers see no new values.
- **Sentinel stripped** from data forwarded to renderer before `onCliExited` fires.
- **Pane not torn down** on cli-exited (`alive` stays `true`, no `forget()`/`kill()` call).
- **Notification**: `rpc-router.ts` wires `onCliExited` to call `pushPtyExitNotification`, identical to the direct-mode `onPaneEvent` path.

### Phase 3 (this packet — per-pane safe-scope dispatch, flag default 'direct')

**Scope**: make `assistant.dispatchPane` (Sigma/Jorvis) work correctly in shell-first mode
via graceful per-pane fallback — NOT a racy timing fix.

**Provider taxonomy by prompt-delivery path:**
- Path A (arg injection via `oneshotArgs`): claude (`-p`), codex (`-q`)
- Path A (arg injection via `initialPromptFlag`): gemini (`-i`)
- Path B (post-spawn `pty.write` 600 ms after spawn): kimi, opencode

**Path A** works in shell-first without changes: the prompt is part of the CLI argv that
`spawnLocalPty` writes into the shell as a single command line. No race.

**Path B** has a race in shell-first: the 600 ms `pty.write` may hit the shell
before the CLI has launched. Fix: in `executeLaunchPlan`, compute `effectiveSpawnMode`
via `effectivePaneSpawnMode()` (exported pure helper in `workspaces/launcher.ts`).
When global mode is `'shell-first'` AND a pane has `initialPrompt` AND its provider
has neither `oneshotArgs` nor `initialPromptFlag` → that pane's effective spawn mode
is `'direct'`. It loses shell-durability for that launch, but the prompt lands
correctly. Other panes in the same workspace keep shell-first.

**Deferred enhancement**: a "CLI-ready signal" approach (e.g. write the prompt only
after the CLI process has fully started inside the shell, detected via a shell hook or
sentinel output) would give Path B providers both prompt delivery AND shell-durability.
This is intentionally left as a future enhancement and must NOT be implemented as a
timing hack.

**CRITICAL INVARIANT preserved**: when global mode is `'direct'`, `effectivePaneSpawnMode`
always returns `'direct'`; the override path never fires; behaviour is byte-for-byte
identical to pre-Phase-3.

**Implementation location**: `app/src/main/core/workspaces/launcher.ts`
- `effectivePaneSpawnMode()` (exported pure helper, ~8 lines, fully testable)
- `executeLaunchPlan()` — replaces the 3-line inline override with the helper call

**Tests**: `app/src/main/core/workspaces/launcher.test.ts`
- 11 new cases covering all combinations of global mode × provider path × initialPrompt
