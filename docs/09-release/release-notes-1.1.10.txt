SigmaLink v1.1.10 — Reliability hotfix from Gemini audit

Six P1 reliability bugs found by a parallel Gemini CLI + 10-agent
Ruflo audit, plus four perf wins and a dead-code sweep. Fixed by a
4-coder Ruflo swarm working on disjoint file scopes. Zero
behavioural changes for the happy path; the wins are all in failure
modes (process leaks, race conditions, broadcast aborts, animation
waste, hung child kills, dead module deletions).

Why v1.1.10 exists
==================

Gemini's audit at `app/docs/investigation/codebase-audit-v1.1.3.md`
identified critical reliability gaps across three layers. Each was
real, reproducible, and would degrade UX under sustained use. v1.1.9
shipped perf paired refactor + lint zero but did not address these
reliability findings. v1.1.10 closes them.

Provider registry cleanup (Claude/Codex/Gemini/OpenCode/+Kimi) is
NOT in this release — it gets its own scope as v1.1.11.

What v1.1.10 ships
==================

Backend reliability (3 P1 bugs)
-------------------------------

1. **`resolveAndSpawn` fallback now actually tries alternatives**
   (`src/main/core/pty/local-pty.ts`). Previously when a command
   ENOENTed, `spawnLocalPty` swallowed the error and returned null
   (or surfaced via async child-exit-127 on POSIX), so the
   launcher's fallback loop terminated without trying the next
   candidate. v1.1.10 adds a synchronous `resolvePosixCommand`
   PATH pre-flight inside `spawnLocalPty` that throws a real
   `ENOENT` (with `.code = 'ENOENT'`, `.errno = -2`,
   `.syscall = 'spawn'`, `.path`) — the launcher's existing
   `isENOENT()` heuristic + fallback walk now find the right
   binary. Smaller-diff than refactoring the result-type contract.

2. **`pty.forget()` now kills the underlying process**
   (`src/main/core/pty/registry.ts`). Previously `forget()` just
   removed the entry from the registry Map, leaking the underlying
   PTY child as a ghost process. v1.1.10 SIGTERMs the live PTY +
   arms a single `setTimeout(5000)` SIGKILL fallback (`.unref()`'d
   so it never blocks shutdown). New `isProcessAlive(pid)` helper
   probes via `process.kill(pid, 0)` (handles `EPERM` correctly).

3. **`killAll()` now uses ONE 5s timer regardless of session count**
   Previously scheduled a fresh setTimeout per live session (N
   sessions → N timers). With 16 panes that was 16 timers wasting
   timer-pool slots. Now: collect all survivor PIDs in one pass,
   SIGTERM each via `pty.kill()`, then ONE 5s `setTimeout` SIGKILLs
   stragglers. Verified via `vi.getTimerCount()` in the regression
   test.

4. **`execCmd` kills ghost child on maxBuffer overflow**
   (`src/main/lib/exec.ts`). Previously when stdout/stderr bytes
   exceeded `maxBuffer`, Node returned `ERR_CHILD_PROCESS_STDIO_MAXBUFFER`
   but the child kept running in the background. v1.1.10 adds
   `handleOverflow()` that destroys both stdio streams, sends
   SIGTERM, and arms a 5s SIGKILL fallback (shared with the
   `timeoutMs` path — no double timers). New `maxBufferExceeded:
   boolean` field on `ExecResult` (additive; no caller breakage).

Orchestration reliability (3 P1 bugs)
-------------------------------------

5. **Mailbox broadcast no longer aborts on per-recipient failure**
   (`src/main/core/swarms/mailbox.ts`). Previously the broadcast
   loop for `@all` / `@coordinators` had a single try/catch
   wrapping the entire JSONL-mirror + paneEcho pass. One destroyed
   PTY mid-broadcast → all subsequent agents stranded without the
   message. v1.1.10 wraps each recipient's mirror + echo in
   per-recipient try/catch + `console.warn` with `swarmId` +
   failing agent key. Remaining recipients still receive.

6. **`addAgentToSwarm` role index is now atomic**
   (`src/main/core/swarms/factory.ts`). Previously a non-atomic
   read-then-write: `count = SELECT ... FROM swarm_agents WHERE
   role = ?`; `INSERT ... role_index = count`. Concurrent
   same-role adds computed identical `roleIndex` → DB unique
   constraint violation. v1.1.10 wraps the count + max(roleIndex)
   + INSERT inside `getRawDb().transaction(() => { ... })` so
   concurrent same-role adds serialize on SQLite's write lock and
   produce contiguous indices. The 20-cap MAX_SWARM_AGENTS check
   stays inside the txn so the 21st-agent refusal still fires.

7. **StdinWriter queue has per-write timeout**
   (`src/main/core/assistant/runClaudeCliTurn.emit.ts`). Previously
   if the CLI child stopped reading stdin (hung, crashed, dropped
   the pipe), the promise chain hung indefinitely. v1.1.10 adds
   `STDIN_WRITE_TIMEOUT_MS = 30_000` racing every enqueued write;
   on timeout, rejects with `Error('stdin_write_timeout')` and
   invokes a configurable `onTimeout` callback. The turn driver
   passes `onTimeout: () => child.kill('SIGTERM')` so the child
   doesn't outlive its broken stdin. Synchronous `child.stdin.write`
   throws are now caught + surfaced as rejections instead of
   escaping to unhandledRejection.

Frontend perf (4 wins)
-----------------------

8. **Terminal, Sidebar, Launcher migrated to per-slice subscription**
   v1.1.9 introduced `useAppStateSelector<T>` and migrated 4 hot
   rooms (CommandRoom, CommandPalette, SwarmRoom, OperatorConsole).
   Gemini found Terminal, Sidebar, and Launcher still on full
   `useAppState()` — re-rendering on every dispatch. v1.1.10:

     * Terminal subscribes to ONE slice: `state.activeWorkspace?.id`.
     * Sidebar uses 5 independent slice selectors (`activeWorkspace`,
       `sidebarCollapsed`, `openWorkspaces`, `workspaces`, `sessions`)
       + `useAppDispatch`.
     * Launcher uses 2 outer selectors + `useAppDispatch`; the
       inner `BottomActionRow` doesn't subscribe to state at all.

   Result: dispatching unrelated actions (chat messages, swarm
   broadcasts, browser state, notifications) no longer triggers
   re-renders in any of the three. Verified via `useSyncExternalStore`
   + `Object.is` snapshot identity guards.

9. **Constellation animation gated by visibility**
   (`src/renderer/features/operator-console/Constellation.tsx`).
   Previously the force-directed graph's `requestAnimationFrame`
   loop ran continuously even when the OperatorConsole tab was
   hidden or the user navigated to a different room. v1.1.10
   gates the rAF loop on dual signals: Page Visibility API +
   IntersectionObserver. Loop pauses when either reports hidden,
   resumes when both report visible. Cleanup disconnects observer
   + removes visibilitychange listener on unmount. 6 new tests
   cover start/pause/resume/cleanup paths.

Dead code (-82 LOC)
--------------------

10. **`PhasePlaceholder.tsx` + `placeholders/` dir deleted**.
    Zero callers across `src/`. Pre-Electron development remnant.

11. **`RoomChrome.tsx` deleted, single caller inlined**.
    Only `SettingsRoom.tsx` used it; the unused props
    (`RoomSkeleton`, `loading`, `errorBanner`, `actions`,
    `loadingHint`) dropped entirely. The header strip + body
    container inlined into SettingsRoom for ~12 LOC vs 66 LOC
    removed.

12. **Survey of `src/renderer/lib/` + `src/main/lib/`**: no other
    zero-caller modules found. All 10 lib files have ≥1 importer.
    File-level dead-code surface is now exhausted.

Upgrade path
============

Drop the new SigmaLink.app into /Applications. Existing kv state +
workspaces + agent sessions survive. The reliability wins are
felt under failure modes — a crashing CLI used to hang the Sigma
Assistant indefinitely; now it times out after 30s with a clear
error.
