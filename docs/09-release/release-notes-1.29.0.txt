SigmaLink v1.29.0 — 2026-05-26
==============================

Operator smoke fixes (SF-1..SF-6). The real-device pass after v1.25–v1.28
surfaced six bug/polish items; this release closes all of them.

Three worktree-isolated lanes (1 Opus + 2 Sonnet), each verified its root
cause before writing a line of fix, plus a lead-owned notification-sound
widening and an SF-2 single-source-of-truth fix -> full gate in main.

Headline: SF-2's "malformed --resume" was NOT arg-mangling. The shell-first
command builder was clean all along (disproven against the real claude 2.1.150
binary — the FULL session id reached claude). The real cause was a
cwd -> project-slug computation bug.


Fixed
-----

- SF-1  Telegram replies sent twice.
    core/remote/bridge.ts relayed BOTH the debounced delta-accumulated buffer
    AND the final payload (identical text) -> duplicate message. onAssistantState
    is now a FINAL-ONLY relay: delta accumulates silently, final cancels any
    stale timer and flushes exactly once; error-only turns keep the debounced
    path so they still reach the operator. (Single assistant:state subscription
    confirmed — not a double-subscribe.)

- SF-2  "No conversation found" resuming a Claude session in a fresh workspace.
    claude resolves `--resume <id>` strictly by a cwd-derived project-dir slug,
    and its slug rule replaces EVERY non-alphanumeric char with '-'.
    claudeSlugForCwd only replaced '/', so any cwd containing a space/dot/paren
    — e.g. the macOS "Library/Application Support/…" path the worktrees live
    under — symlinked the session JSONL into the WRONG project dir, and the
    resume (running in the real cwd) returned the operator's error.
    Fix: slug = cwd.replace(/[^a-zA-Z0-9]/g, '-'). Also pointed the session-
    picker disk scanner (session-disk-scanner.ts — identical latent '/'-only
    copy) at the shared helper for a single source of truth.

- SF-3  `1;2c` typed into panes on window focus-switch.
    A program in the PTY emits a Device-Attributes query (ESC[c) on OS focus-
    regain; xterm answers via onData with ESC[?1;2c, and the keystroke pipe
    forwarded that answer back into the PTY (the shell echoed the printable
    "1;2c"). Added stripDeviceAttributesResponses() to the onData path,
    filtering ONLY DA Primary (CSI ?…c) and Secondary (CSI >…c) replies.
    Cursor-Position (…R) and Device-Status (…n) reports — which programs
    legitimately consume — are preserved (no introducer/final-byte collision).

- SF-4  Notifications dropdown unstyled on the Glass theme.
    The dropdown container had bg-popover but was missing `sl-glass relative`,
    so the .sl-glass::before specular surface never rendered (text overlapped
    the transparent surface). Added `sl-glass relative` (matching RightRail).

- SF-5  No sound on new notifications.
    Not a regression — the tone was deliberately gated to warn/error/critical,
    so routine info-level events (agent/swarm chatter) stayed silent. The
    operator's notifications were info-level. Per operator decision, WIDENED to
    all severities: a tone now fires on any new unread notification, still
    respecting the notifications.sound toggle (default ON).

- SF-6  Right-rail Jorvis panel squashed against the window edge.
    Added px-3 to the JorvisRoom rail-variant content column (the Orb +
    "Ask Jorvis…" input were flush to the rail border). Standalone variant
    unchanged.


Not in this release (tracked follow-up)
---------------------------------------

- SF-7  Ruflo MCP daemon auto-init / auto-trust on workspace open. Auto-init
        already runs by default (factory.ts spawns the per-workspace HTTP
        daemon + writes the MCP config unless kv ruflo.autowrite.mcp='0'), but
        (a) spawn silently falls back to stdio if the binary is missing with no
        surfaced status, and (b) the CLI's own /mcp trust gate isn't pre-
        approved. Queued for v1.30.0.

- SF-8  Launch-time Yolo/Bypass mode for CLI panes. The per-provider bypass
        flags already exist (autoApproveFlag), but autoApprove is only wired at
        the swarm-agent level (RoleRoster "Auto" chip); plain panes have no
        toggle. Queued for v1.30.0.

Both documented in docs/03-plan/WISHLIST.md.
