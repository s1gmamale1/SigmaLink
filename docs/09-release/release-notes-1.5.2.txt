🚨 CRITICAL HOTFIX — cross-machine sync renderer surface restored
-----------------------------------------------------------------

**If you tried to use cross-machine sync via the UI on v1.5.0 and saw
error banners / non-functional Settings → Sync / SetupWizard stuck on
load, this release fixes that.**

Root cause: v1.5.0 packet 09 shipped the entire cross-sync backend
(libsodium e2ee, HLC, isomorphic-git transport, BIP-39 mnemonic, all
8 controller methods) but forgot to add the `sync.*` IPC channels to
the `CHANNELS` allowlist in `app/src/shared/rpc-channels.ts`. The
preload bridge hard-rejects unlisted channels, so every `rpc.sync.*`
call from the renderer (status, isConfigured, listConflicts, disable,
enable, exportMnemonic, resolveConflict, recoverFromMnemonic) was
rejected at the preload layer with an error banner. Settings → Sync
showed error UI; SetupWizard couldn't progress; SyncTab's pending-
upgrade badge was unreachable.

Why it slipped through v1.5.0 tests:
- SetupWizard.test.tsx mocks the rpc layer entirely (bypasses preload).
- No e2e smoke test exercises sync paths.
- The test gap is added to the v1.5.3 backlog (CHANNELS-vs-AppRouter
  cross-reference test + e2e sync smoke).

If you set up cross-sync between v1.5.0 release and v1.5.2 release
and saw no UI errors, you weren't actually syncing — the renderer was
hard-failing every call. Upgrade and re-attempt setup; encrypted blobs
already on your git remote (if any) will decrypt correctly on the next
pull (backward compat preserved).

Upgrade notes
-------------

If you set up cross-sync in v1.5.0 and saw the renderer fail silently:
no data loss occurred. The backend wrote nothing because the renderer
never successfully called `enable()`. Upgrade to v1.5.2, run the setup
wizard again, and your sync will work this time.

If you already had a working cross-sync setup (somehow — possibly via
direct kv manipulation or a corner case), your existing blobs decode
correctly on the next pull.
