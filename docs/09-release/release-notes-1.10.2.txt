Sigma/Jorvis dispatch works in shell-first mode (all providers)
---------------------------------------------------------------

assistant.dispatchPane spawns panes with an initialPrompt. Three
prompt-delivery mechanisms exist; Phase 3 makes all correct under
shell-first:

- initialPromptFlag (gemini -i) + oneshotArgs (claude -p, codex -q):
  prompt is a CLI arg → Phase 1 shell-first injection already writes
  `<cli> <flag> "<prompt>"\n` to the shell. These panes stay
  shell-first. Verified + tested.
- Path B / stdin (kimi, opencode — no prompt flag): prompt delivered
  by post-spawn pty.write, which would RACE the shell→CLI startup in
  shell-first. Rather than a half-built timing fix, Phase 3 applies a
  clean PER-PANE FALLBACK: when global pty.spawnMode is 'shell-first'
  but a pane has initialPrompt AND its provider has neither prompt
  flag, that pane spawns 'direct' (effectivePaneSpawnMode() in
  local-pty.ts, applied in executeLaunchPlan). Prompt delivered
  correctly (no race); that pane forgoes shell-durability — a
  documented, fully-functional degradation. Other panes keep
  shell-first.

The "stdin-prompt + shell-durability via a CLI-ready signal"
combination is a deliberately-deferred future enhancement (not
half-built here).

Zero regression at default: effectivePaneSpawnMode('direct', …)
always returns 'direct'; the per-pane override fires only when global
mode is 'shell-first'. Smoke e2e boots at default.
