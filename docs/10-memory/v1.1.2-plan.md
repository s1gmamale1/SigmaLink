## Context — what's broken in v1.1.1

User dogfooded v1.1.1 (commit `0262383`) and surfaced four real defects, all of which converge on one architectural gap: **the assistant emits intent but the host bridge doesn't dispatch it**.

1. **`launch_pane` tool fires but doesn't spawn a PTY** (`BUG-V1.1.1-01`). Sigma's Tool calls panel shows `{ name: "launch_pane", args: { workspaceRoot, provider, count, initialPrompt }, result: { fromCli: true, input: {…}, result: {…} }}` — the CLI emitted the tool call, the renderer traced it, but `runClaudeCliTurn.ts:routeToolUse()` only feeds the envelope into `ToolTracer` (for visualisation) — it never calls `controller.invokeTool('launch_pane', args)` to actually do the thing. Same gap for `dispatch_pane`, `dispatch_bulk`, `create_swarm`, `ref_resolve`, every tool in `tools.ts`.

2. **Sigma can't enumerate live sessions** (`BUG-V1.1.1-02`). User asked "do you see how many agents currently launched in our workspace?" and Sigma replied "No active swarms or panes…" while 4 agents were visibly running. `buildSigmaSystemPrompt()` reads from the DB at turn start; freshly-spawned PTYs that haven't yet flushed to disk aren't visible. Worse: even if the DB read was perfectly current, the system prompt is built ONCE per turn — there's no way for Sigma to refresh mid-turn.

3. **Inter-agent broadcast inert** (`BUG-V1.1.1-03`). Operator side-chat broadcasts to `@all` / `@coordinators` land in the operator log but every agent shows `0 msgs`. Likely regression from the rc3 cross-swarm-leak fix (`BUG-V1.1-02-IPC`) tightening swarm-id scoping too aggressively.

4. **Ruflo MCP not auto-connected for spawned agent CLIs** (`BUG-V1.1.1-04`). Every claude/codex/gemini pane that SigmaLink spawns reports "RuFlo is not initialized" or shows Ruflo MCP as disconnected. Each agent reads its own per-tool MCP config; SigmaLink's embedded Ruflo daemon isn't discovered. User's exact phrasing: "do I have to manually connect the Ruflo MCP running in this directory again for any new agent? […] But that kills the point of Ruflo".

**Outcome**: ship `v1.1.2` where Sigma Assistant can (a) actually launch and dispatch panes, (b) see and reason about live workspace state, (c) coordinate with other agents that share its memory.

## Recommended approach

### Step 1 — Tool dispatch parity (L, ~3d) — THE BIG ONE

Make every `tool_use` envelope the CLI emits actually execute against the SigmaLink main process, with the result fed back as a `tool_result` envelope on the next CLI turn.

**Architecture**: today `runClaudeCliTurn.ts:routeToolUse()` only traces. After this step, it routes to `controller.invokeTool(name, args)` for every Sigma-known tool name, captures the result, and writes a `tool_result` envelope back to the CLI's stdin. The CLI then continues its turn with the actual result in scope.

**Tool-use protocol — exact JSON shapes (the bit Codex must get right)**:

The Claude Code CLI in `--output-format stream-json --verbose` mode emits and accepts envelopes that match Anthropic's Messages API tool-use shape. The CLI writes `tool_use` blocks INSIDE `assistant` envelopes on stdout, then blocks waiting for `user` envelopes on stdin with matching `tool_result` blocks.

Stdout from the CLI (parse — already handled by `cli-envelope.ts`):
```json
{"type":"assistant","message":{"id":"msg_X","role":"assistant","content":[
  {"type":"text","text":"I'll launch a codex pane for you."},
  {"type":"tool_use","id":"toolu_abc123","name":"launch_pane","input":{"workspaceRoot":"/path","provider":"codex","count":1,"initialPrompt":"Give 1 sentence introduction of yourself."}}
]}}
```

What we write to stdin (NEW — this is the work):
```json
{"type":"user","message":{"role":"user","content":[
  {"type":"tool_result","tool_use_id":"toolu_abc123","content":"{\"sessionId\":\"abc\",\"provider\":\"codex\",\"paneIndex\":4}"}
]}}
```

Notes:
- `tool_use_id` MUST match exactly. The CLI rejects orphan results.
- `content` is a STRING (stringify the JSON result). Some CLI builds also accept content as an array `[{"type":"text","text":"..."}]` — stringified-JSON is the safer common denominator.
- One stdin line per tool result, newline-terminated. The CLI's JSONL parser splits on `\n`.
- For errors: same envelope shape, set `is_error: true` and put `{"error":"<msg>"}` in content: `{"type":"tool_result","tool_use_id":"toolu_X","is_error":true,"content":"{\"error\":\"unknown tool\"}"}`.

**Current `routeToolUse` signature** (from `runClaudeCliTurn.ts`):
```ts
function routeToolUse(
  envelope: CliAssistantEnvelope,
  deps: CliTurnDeps,
  turn: CliTurnHandle,
): void { /* trace only */ }
```

**New signature** (add the dispatcher):
```ts
async function routeToolUse(
  envelope: CliAssistantEnvelope,
  deps: CliTurnDeps,
  turn: CliTurnHandle,
  child: CliChildLike,
  dispatchTool: (name: string, args: unknown) => Promise<unknown>,
): Promise<void> { /* trace + dispatch + stdin write */ }
```

**Edits**:
- `app/src/main/core/assistant/runClaudeCliTurn.ts:routeToolUse()` — extend signature as above. For every `tool_use` content block in the envelope:
  1. Trace it via existing `ToolTracer` (keep).
  2. Lookup `name` in `tools.ts` registry. If unknown → skip to step 4 with `{error: "unknown_tool", name}`.
  3. `try { result = await dispatchTool(name, args); } catch (e) { result = { error: e.message }; isError = true; }`.
  4. Build the `tool_result` envelope `{ type: "user", message: { role: "user", content: [{ type: "tool_result", tool_use_id: block.id, content: JSON.stringify(result), is_error: isError }] }}`.
  5. Write to `child.stdin` as one `\n`-terminated line. Use a serialised write queue (single-flight `Promise` chain) so concurrent tool_use blocks within one envelope are written in order without race.
- **Streaming-safe write queue** (~20 lines): `let writeChain = Promise.resolve(); function enqueueWrite(line: string) { writeChain = writeChain.then(() => new Promise<void>((resolve, reject) => { child.stdin.write(line, (err) => err ? reject(err) : resolve()); })); return writeChain; }`. Drain on `close`.
- `app/src/main/core/assistant/controller.ts` — pass `dispatchTool` into `runClaudeCliTurn` as a closed-over function that calls the existing `invokeTool()` (line 189-259) with proper auth + workspace context. The closure has access to `conversationId`, `workspaceId`, so per-turn permissions still work.
- `app/src/main/core/assistant/tools.ts` — audit the 10 canonical tools; for each, verify args shape matches what Claude CLI sends (capability matrix: `launch_pane`, `dispatch_pane`, `dispatch_bulk`, `create_swarm`, `ref_resolve`, `list_active_sessions`, `list_swarms`, `list_workspaces`, `memory.search`, `memory.create`). Add the three `list_*` tools (Step 2 covers their bodies).
- `app/src/main/core/assistant/runClaudeCliTurn.test.ts` — add 4 new test cases (use existing `spawnOverride` fixture to inject a fake child with controllable stdin/stdout):
  - `tool_use launch_pane → dispatchTool called → tool_result line written to stdin → assert stdin contents`
  - `tool_use unknown_tool → tool_result with is_error:true → CLI continues`
  - `dispatchTool throws → tool_result envelope with error body → CLI sees the failure`
  - `multiple tool_use in one assistant envelope → all dispatched, results written in input order`
- `app/tests/e2e/assistant-cli-launch-pane.spec.ts` (new) — Playwright e2e: ask Sigma "launch 1 codex pane with a 1-sentence intro", wait up to 30s, assert a codex PTY appears in Command Room with the intro prompt typed. Skip when `claude --version` is absent.

**Risk**: medium. The dispatch loop must NOT deadlock — the CLI's stdin can buffer, the dispatchTool can be slow (long-running tool calls), and the CLI might already be partway through emitting the next envelope. Mitigation: the serialised write queue above prevents concurrent stdin.write; for slow tools (`create_swarm` provisions worktrees, 3-10s), no special handling needed because the CLI blocks waiting for the result anyway. Add a `dispatchTool` 30s timeout that resolves with `{error: "tool_timeout"}` so a hung handler doesn't pin the turn forever.

### Step 1b — Auto trajectory wrap for SONA learning (S, ~0.5d) — fold into Step 1

Sigma Assistant should learn from its own task outcomes via Ruflo's trajectory system, the same way Claude (me) benefits from the per-prompt `[INTELLIGENCE]` hook. ~30 lines.

**Edits inside `runClaudeCliTurn.ts`** (combined PR with Step 1):
- Before invoking the CLI: `const trajectoryId = await deps.ruflo?.trajectoryStart({ task: prompt.slice(0, 200), namespace: 'sigma-assistant' });` (fail-soft if ruflo unavailable).
- On every `tool_use` dispatched: `await deps.ruflo?.trajectoryStep({ trajectoryId, action: name, observation: JSON.stringify(args).slice(0, 500) });`.
- On `result` envelope (success): `await deps.ruflo?.trajectoryEnd({ trajectoryId, verdict: 'success', summary: result.text.slice(0, 300) });`.
- On error/cancel: `await deps.ruflo?.trajectoryEnd({ trajectoryId, verdict: 'failure', summary: errorMsg });`.
- Wire the calls via the existing Ruflo proxy at `app/src/main/core/ruflo/proxy.ts` (extend with `trajectoryStart/Step/End` if not yet exposed; thin wrappers around `mcp__ruflo__hooks_intelligence_trajectory-*`).
- 1 new test in `runClaudeCliTurn.test.ts`: assert trajectoryStart/End fire with correct verdicts for success and failure paths.

This is what gets Sigma Assistant the same cross-session learning Claude Code benefits from. After ~50 trajectories, the assistant starts surfacing "similar past task" suggestions automatically via the existing pattern-search hook in BridgeRoom.

## Open risks

- **R-1.1.2-1**: Tool dispatch round-trip latency. If a tool takes 5+ seconds (e.g., `create_swarm` provisions multiple worktrees), the CLI may time out waiting for `tool_result`. Mitigation: emit a streaming "tool_use_in_progress" envelope back to the renderer (not the CLI) so the user sees "working…", and ensure the CLI's tool-result-wait timeout is generous (the CLI does block on stdin by default for tool results).
- **R-1.1.2-2**: `mcp-autowrite` may surprise users who manually edited their `.mcp.json` / `.codex/config.toml`. Mitigation: only WRITE if no `ruflo` entry exists OR if existing entry has identical command. Never clobber.
- **R-1.1.2-3**: `BUG-V1.1.1-03` may turn out to be a deeper protocol issue (e.g., the rc3 cross-swarm fix needs to be reverted or refined). Time-box Step 4 at 4 hours; if not solved, defer to v1.1.3.
- **R-1.1.2-4**: Live `list_*` tools may surface stale data if the registry races with PTY spawn. Use the same kv pattern as the v3 mailbox: serialised reads via a single accessor.
