# Plan ‚Äî v1.1.3: multi-workspace, pane resume, add-pane, Ruflo pre-flight

> v1.1.2 made Sigma Assistant actually dispatch tools and persist workspace+room across restart. v1.1.3 takes the next jump: multiple workspaces live simultaneously, all reopen on restart, panes restore via each CLI's `--resume`, the user (and Sigma) can grow a swarm after creation, and every workspace open eagerly verifies Ruflo is ready for the CLIs it spawns.

## Context ‚Äî why v1.1.3 exists

User dogfooded v1.1.2-rev3 (commit `c1452bd`, arm64 DMG launches clean, Sigma Assistant can read+write panes via the new MCP host server). Three real-world gaps surfaced during real work:

1. **"BRIDGE" still shows in the chat role label.** v1.1.1 rebranded sidebar / right-rail tab / placeholders / EmptyState etc. to "Sigma Assistant", but `ChatTranscript.tsx` still tags assistant messages with the literal label "BRIDGE". Last user-facing string left in the rebrand.

2. **Single-workspace model is wrong.** Today `state.activeWorkspace: Workspace | null` is the only opened concept. Switching workspaces in the sidebar discards the prior one's runtime state (sessions stay alive in DB but the renderer treats them as inert). On quit, only the active workspace + room are persisted (v1.1.2 minimum-viable session-restore). User wants: open many workspaces in parallel like browser tabs; switching keeps each alive; restart restores all of them.

3. **Pane state evaporates across restart.** `agent_sessions` rows survive in DB but the actual claude/codex/gemini processes don't (PTYs die when SigmaLink quits). v1.1.2 left `resumeArgs: ['--resume']` declared in the provider registry but UNUSED ‚Äî no code consumes it, no DB column captures the CLI's per-session id, no restore-on-boot launcher. User has to manually re-spawn every agent.

4. **Swarm count is locked at creation.** The Swarm Wizard's preset (5 / 10 / 15 / 20 panes) is enforced at create-time. No UI to add an agent post-launch, even though `swarms.factory` would happily accept one. Sigma Assistant can't grow a swarm either ‚Äî `launch_pane` exists but `add_to_swarm` doesn't.

5. **Ruflo bootstrap is lazy + verifies nothing.** Today the Ruflo supervisor starts ONLY when the user clicks "Download Ruflo" in Settings OR on the first `ruflo.*` RPC call. Workspace open writes `.mcp.json` etc. blindly with no verification that the spawned CLIs will actually discover the server. Result: every workspace creation requires manual reassurance that Ruflo is wired.

**Outcome**: ship `v1.1.3` with: chat label rebrand complete, multi-workspace tabs that survive switching + restart, pane auto-resume on boot, growable swarms (UI + Sigma tool), and a pre-flight Ruflo check that catches breakage before the user spawns agents.

## Locked design decisions (from AskUserQuestion 2026-05-11)

- **Workspace cap**: 8 sidebar tabs + overflow drawer. Runtime unlimited.
- **Pane resume**: auto-respawn silently with `--resume <id>`. Failures fall to empty pane state without prompting.
- **Sigma Assistant scoping**: shared right-rail panel; switching workspace swaps active conversation. Conversations are already DB-keyed by workspace, so no data leak.
- **Ruflo verification depth**: fast (process `ready` + autowrite succeeded) by default; "Strict MCP verification" toggle in Settings ‚Üí Ruflo opts into the thorough handshake.

### Step 3 ‚Äî Pane resume via CLI `--resume <id>` (L, ~3d)

Restore agent panes on app restart by extracting each CLI's session id from its early output, persisting to DB, then re-spawning with `--resume <id>` on next boot. Today this is dead code in `providers.ts` ‚Äî wire it.

**Backend (main)**:
- New migration `app/src/main/core/db/migrations/0011_agent_session_external_id.ts` ‚Äî adds `external_session_id TEXT` to `agent_sessions`. Idempotent ALTER TABLE inside BEGIN/COMMIT/ROLLBACK (mirror migration 0010 pattern).
- New `app/src/main/core/pty/session-id-extractor.ts` (~120 lines):
  - For each provider, a regex / JSONL parser that extracts the session id from the first ~50 lines of PTY output.
  - claude code: parses `system` ‚Üí `init` JSONL envelope from the `--output-format stream-json --verbose` mode for non-interactive turns; for interactive PTY, parses the welcome banner `Session: <uuid>` line (verify against real claude CLI banner).
  - codex: parses `codex resume <id>` syntax ‚Äî find the "Session ID: <id>" banner emitted by codex on startup.
  - gemini: investigate whether gemini emits a session id at all; if not, mark provider as "not resumable" and skip.
  - Returns `{ sessionId, providerId, raw }`.
- `app/src/main/core/pty/registry.ts` ‚Äî when PTY first emits data, run the extractor; on first hit, write `external_session_id` to `agent_sessions` via the existing controller. Stop scanning after either a hit or 100 lines (whichever first) so we don't waste CPU on long-running output.
- New `app/src/main/core/pty/resume-launcher.ts` (~150 lines):
  - On app boot (`createWindow` ‚Üí after restore), enumerate every `agent_sessions` row WHERE `workspace_id IN <restored set>` AND `status='running'` AND `external_session_id IS NOT NULL`.
  - For each, call `resolveAndSpawn` with `extraArgs: [...provider.resumeArgs, externalSessionId]` (consuming the existing `resumeArgs` field).
  - Update row `status='running'` + new PID + new started_at.
  - Failures (CLI returned non-zero, missing session, --resume unsupported): set `status='exited'` with error code, emit toast "Could not resume <provider> session <id>".

**API**:
- New RPC `panes.resume(workspaceId)` ‚Äî batch-resume all eligible rows for a workspace. Called by Step 6 on boot.
- Allowlist + zod schema.

**Frontend**:
- `app/src/renderer/features/command-room/PaneHeader.tsx` ‚Äî small "resumed" badge for first 5s when a pane is restored (visual feedback).
- No new flows; resume is silent per user choice.

**Tests**:
- `pty/session-id-extractor.test.ts` (NEW) ‚Äî fixtures for claude / codex / gemini early output; assert extracted id.
- `pty/resume-launcher.test.ts` (NEW) ‚Äî mock PTY registry, verify `resumeArgs` flag is appended; resume failure path doesn't crash.
- E2E: open workspace, spawn 2 agents, quit app, relaunch ‚Äî assert 2 panes return.

**Docs**:
- New `docs/04-design/pane-resume-protocol.md` (~80 lines) ‚Äî per-CLI session id extraction strategy + fallback behaviour.

### Step 5 ‚Äî Ruflo bootstrap pre-flight + per-CLI verification (M, ~1.5d)

Eager-start Ruflo on workspace open; verify the autowrite landed; surface per-workspace readiness.

**Backend (main)**:
- `app/src/main/core/ruflo/supervisor.ts` ‚Äî new method `ensureStarted()`: idempotent ‚Äî if state is already `ready`, no-op; else kick off `start()` + await up to 5 seconds for `ready` event.
- `app/src/main/core/workspaces/factory.ts:openWorkspace()` ‚Äî after `writeWorkspaceMcpConfig(abs)`, call `rufloSupervisor.ensureStarted()`. Don't block workspace creation on Ruflo readiness (Ruflo failure is degraded UX, not blocker).
- New `app/src/main/core/ruflo/verify.ts` (~120 lines):
  - `verifyForWorkspace(workspaceRoot, mode: 'fast' | 'strict')`:
    - Fast: read back each per-CLI config file (`<root>/.mcp.json`, `~/.codex/config.toml`, `~/.gemini/settings.json`), assert `ruflo` entry present + `command === 'npx'`. ~50ms.
    - Strict: spawn a transient probe per CLI ‚Äî `claude mcp list --workspace <root>` / `codex mcp list` / `gemini /mcp list` (or equivalent) ‚Äî parse output, assert `ruflo` appears. ~3-5s per CLI.
  - Returns `{ claude: bool, codex: bool, gemini: bool, errors: [{cli, message}] }`.
- Gate strict mode on `kv['ruflo.strictMcpVerification']` (default off).

**API**:
- New RPC `ruflo.verifyForWorkspace(workspaceRoot)` ‚Äî returns the verification result.
- Event `ruflo:workspace-verified` per workspace ‚Äî pushed once on workspace open.
- Allowlist + zod.

**Frontend**:
- New `app/src/renderer/components/RufloReadinessPill.tsx` (~60 lines) ‚Äî small status pill rendered in the workspace breadcrumb area. Three states: green (verified), amber (partial ‚Äî some CLIs failed), red (Ruflo unavailable).
- `app/src/renderer/features/settings/RufloSettings.tsx` ‚Äî new "Strict MCP verification" toggle wired to `kv['ruflo.strictMcpVerification']`.

**Tests**:

## Open risks
